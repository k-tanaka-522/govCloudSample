# Fable 5 業務利用評価キット：架空行政RFPベンチマーク 指示書一式

目的：架空の行政RFPを題材に、Claude Code（Fable 5）が「SIerが行政に納品できるレベル」の
アプリケーション＋ドキュメント一式を仕上げられるかを、再現可能なベンチマークとして評価する。
同一RFP・同一採点表で Fable 5 / Opus 4.8 / Sonnet を横並び比較できる構造にする。

---

## 0. ディレクトリ構成と役割分離の原則

```
fable5-eval/
├── 00-framework/          # 評価フレーム（採点表・出題者ノート）※受注エージェントには見せない
│   ├── rubric.md           # 採点ルーブリック（出典必須）
│   ├── traps.md            # 仕掛けた罠の一覧と期待挙動（出題者ノート）
│   └── cost-log.md         # モデル別コスト記録
├── 01-rfp/                 # 受注側に渡す資料（これだけが受注側の世界）
│   ├── rfp.md              # RFP本文
│   ├── attachment-*.md     # 別紙（現行システム概要、データ量、組織体制）
│   └── qa-sheet.md         # 質問票（受注側が記入→発注者役が回答を追記）
├── 02-deliverables/        # 受注エージェントの納品物出力先
│   └── (要件定義書、基本設計書、…、ソースコード、IaC)
├── 03-review/              # 検査官サブエージェントの指摘出力先
│   ├── findings-consistency.md
│   ├── findings-compliance.md
│   └── findings-security.md
└── .claude/
    ├── CLAUDE.md           # 受注側プロジェクトの恒久指示
    └── agents/             # 検査官サブエージェント定義
        ├── inspector-consistency.md
        ├── inspector-compliance.md
        └── inspector-security.md
```

**鉄則（コンテキスト分離）**
1. 受注エージェントは `01-rfp/` と `02-deliverables/` しか読めない前提で動かす。
   `00-framework/`（採点表・罠ノート）は絶対に読ませない。CLAUDE.md に明記する。
2. 検査官は `02-deliverables/` の完成物と `00-framework/rubric.md` だけを読む。
   受注側の作業ログ・思考過程・会話履歴は渡さない（成果物のみで判定させる）。
3. 罠ノート（traps.md）は採点集計時まで誰にも読ませない。人間（出題検収者）専用。

## Phase 0：採点ルーブリック生成（Fable 5 に初稿を作らせる）

新しいセッション（受注側とは別コンテキスト）で実行。

```
あなたは行政情報システム調達の検収専門家です。これから架空の行政RFPに対する
納品物一式を評価するための採点ルーブリックを作成してください。

## 要求事項
1. 根拠の外部固定：各採点項目には必ず出典を付すこと。
   - IPA「共通フレーム2013」の該当プロセス・アクティビティ
   - デジタル庁「ガバメントクラウド」関連文書・地方公共団体情報システム標準化関連文書
   - 政府情報システムにおけるセキュリティ要求（ISMAP等）
   出典が確認できない項目は作らない。制度・標準仕様はWeb検索で現行版を確認し、
   参照した文書名・版・URLを明記すること。あなたの記憶のみに依存した記述は禁止。
2. 評価軸は5カテゴリ：
   A. 成果物の網羅性（共通フレームの成果物一覧との突合）
   B. 文書間整合性・トレーサビリティ（要件ID→設計→テストの追跡可能性）
   C. 制度・標準準拠の正確性（ガバクラ/標準化法/ISMAPの記述が現行情報と一致するか）
   D. 技術的妥当性（AWS設計、IaC規約準拠、セキュリティベースライン）：
      - IaC詳細設計書（Construct/スタック単位）とIaCコード実体の一致
        （CDKファイル名+クラス名レベルでgrep突合可能であること）
      - 基本設計書の構成図 → IaC詳細設計書 → コード実体の3層整合
      - セキュリティ設定（IAM最小権限・SG・暗号化・パブリックアクセス）の
        設計書への明記とコード実体との一致
   E. プロジェクトマネジメント成果物（WBS、課題管理、質問票の運用品質）
   F. 判断成果物の品質（受注者に開放された意思決定の評価）：
      - 技術選定理由書：候補3案以上の比較マトリクス、与件（地域での保守要員確保・
        職員リテラシー）との整合、「技術的最良」と「調達として最良」を区別できているか
      - ADR：重要な設計判断ごとに背景・選択肢・決定・トレードオフ・却下理由が
        記録されているか。ADRなしの重大判断は Block
      - 開発標準書の自己整合性：規約に機械的強制手段（linter/CI設定）が対になっているか、
        かつ実装コードが自分で定めた標準に準拠しているか（乖離は重要度高）
      - フェーズ内分業計画：サブエージェント分業・作業順序の提案が
        根拠を持って構成されているか
      - 体制設計：与件の自社組織を前提にしているか（理想組織の新規設計は減点）、
        責任分界表（RACI）で境界成果物（IaC・パイプライン・監視設定）の所有者が一意か、
        組織制約（インフラ兼任0.5人月等）と運用設計の間の緊張関係を認識し
        対応策（自動化・セルフサービス化）を組み込んでいるか、
        体制とサブエージェント分業の論理が整合しているか
      - リポジトリ戦略：ADRが存在し、体制設計と矛盾しない配置になっているか
      - 監視・運用設計：SLI/SLOが非機能要件から導出されているか、監視がコード化され
        設計書と突合可能か、規模に対して過剰/過少のない適正水準か
        （過剰設計＝教科書的に正しいが調達文脈に不適、も減点対象とする）
3. 各項目には次を付ける：項目ID / 判定基準（何をもって合格か）/
   重要度（Block=検収不可・Warn=条件付き・Info=改善提案）/ 出典 / 採点方法（機械確認可能か目視か）
4. 項目数は初稿で25〜40。網羅より判定基準の明確さを優先。
5. 出力は Markdown の表形式で 00-framework/rubric.md に保存。

このルーブリックは後で人間がレビューします。自信のない項目には [要確認] を付けてください。
```

**人間（あなた）の検収観点**：公的出典が実在するか抜き取り確認 / 現場感覚で抜けている観点
（例：運用引継ぎ、保守性、データ移行のリハーサル）を追記 / Block項目が多すぎないか調整。
赤入れ内容は後で steering に還元する。

## Phase 1：架空RFP生成（罠の仕込み）

ルーブリックとは別セッションで実行。

```
あなたは人口5万人規模の架空の市「○○市」の情報政策課職員です。
以下の条件で業務システム調達のRFP（提案依頼書）と別紙資料を作成してください。

## 題材
地方公共団体の標準化対象20業務から微妙に外れる業務を選ぶこと
（例：公共施設予約管理、補助金交付管理、例規管理 など）。
標準仕様の丸写しでは対応できず、ガバメントクラウドの活用方針・既存標準との
整合判断が必要になる題材が望ましい。

## RFPに含める要素
- 調達の背景・目的（ガバメントクラウド活用、IaCによるモダン化、運用コスト削減）
- 機能要件（20〜30項目、要件IDを付番）
- 非機能要件（性能・可用性・セキュリティ。IPA非機能要求グレードの語彙を使用）
- 納品成果物一覧（IPA共通フレームに基づく文書体系を要求）
- スケジュール、契約条件、提案評価基準
- 別紙：現行システム概要、データ量、職員体制

## 与件の線引き（固定するもの／提案させるもの）
発注者として固定する与件：AWS・ガバメントクラウド利用、IPA共通フレーム準拠の
プロセスと成果物体系、セキュリティベースライン。
受注者の提案に委ねるもの（RFPに明記すること）：プログラミング言語・フレームワーク等の
技術スタック、アプリケーションアーキテクチャ、開発標準、フェーズ内の作業体制。
ただし次の与件を非機能要件・前提条件に自然な形で含めること：
- 「保守・運用要員は地域内で確保できる技術であること」
- 「職員のITリテラシーは標準的であり、特別な訓練を前提としないこと」
（最新流行技術を無批判に選ぶか、調達文脈に合わせて枯れた技術とのバランスを
取れるかを試す与件。罠ノートに期待挙動を記載すること）

## 罠の仕込み（最重要）
次の3種類を1つずつ、自然な文章として埋め込むこと：
1. 矛盾要件：2つの要件が両立しない（例：「個人情報は庁内サーバで保持」と
   「ガバメントクラウド上で完結」）。質問票で照会してくるかを試す。
2. 曖昧要件：定量化されていない非機能要件（例：「十分な応答性能」）。
   定量化の提案・確認をしてくるかを試す。
3. 制度の落とし穴：現行制度を正しく調べないと誤答する論点
   （例：当該業務が標準化対象か否かの判断、ガバクラ利用の対象範囲）。

さらに4つ目の罠として、ネットワーク構成の踏み絵を仕込むこと：
4. 別紙の与件には「本システムの利用者は住民および市職員である」とだけ記載し、
   ネットワーク構成（自治体三層分離のどの系に置くか、職員のアクセス経路、
   ガバメントクラウドとの接続構成）は意図的に明記しない。
   三層分離を自力で論点化（設計判断または質問票での照会）できるかを試す。

罠の内容・位置・期待される正しい挙動は、RFP本文とは別に
00-framework/traps.md（出題者ノート）にまとめること。RFP本文には痕跡を残さない。
traps.md には各罠について「想定される質問」と「発注者の公式回答」を事前に固定して
記載すること（例：罠4への照会には「β'モデル相当を想定」と回答する等）。
検収者（人間）はフェーズ検収・質問票回答の際、この公式回答以外をアドリブで返さない。

出力：01-rfp/rfp.md、01-rfp/attachment-01.md…、00-framework/traps.md
```

**人間の検収観点**：罠が不自然に目立っていないか / 実際の自治体RFPと比べて分量・体裁が
妥当か / 質問票の様式（qa-sheet.md）を用意したか。

## Phase 2：受注側の実行（評価本体）

### 2-1. CLAUDE.md（受注プロジェクトの恒久指示）

```markdown
# プロジェクト：○○市システム調達 提案・構築

## あなたの立場
中堅SIerのプロジェクトチームとして、01-rfp/ のRFPに基づき
提案から納品までを行う。発注者（人間）がフェーズごとに検収する。

## 自社の体制（与件：変更不可。すべての提案はこの体制で実行可能であること）
- アプリケーション開発部：8名（Java / TypeScript 中心。AWS経験は半数、IaC経験者は1名）
- インフラ基盤部：3名（全員が複数案件兼任。本件に割けるのは合計0.5人月/月相当）
- PMO：1名（品質管理・進捗管理。技術判断は行わない）
- 保守運用は納品後、自社の運用チーム（夜間休日は外部委託の一次受付のみ）が担当する
体制設計・リポジトリ戦略・運用設計は、この組織の現実を前提とすること。
理想的な体制を新規に描くことは認めない。組織の制約が技術判断に影響する場合は、
その緊張関係と対応策（自動化・セルフサービス化・教育計画等）を明示すること。

## 読取範囲の制限
- 読んでよいのは 01-rfp/ と 02-deliverables/ のみ。
- 00-framework/ 03-review/ は存在しないものとして扱う。読むことを禁止する。

## プロセス標準
- IPA共通フレーム2013に基づくフェーズゲート方式：
  企画 → 要件定義 → 基本設計 → 詳細設計 → 実装 → テスト → 移行・運用設計
- 各フェーズの成果物を 02-deliverables/ に保存し、発注者の検収を待ってから次へ進む。
  検収前に次フェーズへ進むことを禁止する。
- 全要件に要件ID（REQ-xxx）を付番し、設計書・テスト仕様書まで追跡可能にする。
  トレーサビリティマトリクスを維持・更新すること。

## 詳細設計書フォーマット標準
詳細設計フェーズでは以下の設計書種別ごとに必須記載要素を満たすこと。
フォーマット・記載要素はWeb検索でIPA公開資料・実在のSIer納品事例を確認した上で
採用し、参照元を各設計書冒頭に明記すること。

### 基本設計書との関連付け（全設計書共通）
各詳細設計書のドキュメント管理セクションに以下を必須記載：
- 参照元基本設計書ID
- 対応要件ID（REQ-xxx）
- 対応テストケースID（実装前はTBD可、テスト設計完了時点で埋めること）

### 画面詳細設計書
1. ドキュメント管理（上記共通項目 + 設計書ID / バージョン / 変更履歴）
2. 画面概要・遷移元/遷移先（画面遷移図リンク）
3. 権限・ロール定義
4. 画面項目定義テーブル
   （項目ID / 項目名 / 型 / 桁数 / 必須 / 初期値 / 表示条件）
5. バリデーション仕様（必須 / 桁数 / 形式 / 相関チェック）
6. アクション定義（ボタン/イベント / トリガー / 処理概要 / 遷移先）
7. 呼び出しAPI一覧（API-IDで参照）
8. エラー表示仕様（エラーコード / メッセージ / 表示位置）

### API詳細設計書
1. ドキュメント管理（上記共通項目 + API-ID / バージョン / 変更履歴）
2. エンドポイント / HTTPメソッド / 認証方式
3. リクエスト仕様（ヘッダー / パラメータ / ボディ：型・必須・制約・例）
4. レスポンス仕様（正常系 / 異常系エラーコード一覧）
5. 処理フロー（シーケンス図またはフローチャート）
6. DB操作（参照/更新テーブル一覧 / トランザクション境界）
7. 外部連携一覧
8. 非機能要件（タイムアウト / 冪等性 / レートリミット）
9. 関連クラス/モジュール参照
   （Java/Kotlin/TypeScriptの場合：クラス名・責務・主要依存先を記載。
    クラス図が存在する場合はリンク参照。実装前はTBD可）

### バッチ処理詳細設計書
1. ドキュメント管理（上記共通項目 + バッチID / バージョン / 変更履歴）
2. 起動方式 / スケジュール（cron式）/ 多重起動制御
3. 前提条件・起動条件（先行バッチ / ファイル存在確認等）
4. 処理フロー（フローチャート必須：メイン/エラー分岐を含む）
   （ステップ番号 / 処理名 / 処理内容 / 入力 / 出力）
5. 入出力定義（ファイル/テーブル：レイアウト・件数見込み）
6. DB操作 / トランザクション境界
7. エラーハンドリング
   （エラー種別 / 検知方法 / リトライ有無 / 通知先 / 復旧手順）
8. 非機能要件（処理件数見込み / 処理時間目標 / ログ仕様）
9. 再実行設計（冪等性の担保方法）
10. 関連クラス/モジュール参照（APIと同様の条件）

### クラス/モジュール参照の判断ルール
- Java / Kotlin / TypeScript（クラスベース）→ クラス参照必須
- Python スクリプト系 / シェル → モジュール/関数参照に置き換える
- IaC（Terraform / CDK）→ リソースID参照に置き換える
- 実装完了時点でTBDが残っている場合は検収不可とする

### IaC詳細設計書（CDK）
※受託開発では「動作するコード」に加え、
  発注者が内容を検証できる設計書として以下を整備すること。

1. ドキュメント管理（上記共通項目 + 対応ADR-ID）
2. パラメーター設計（環境別値・変更可否・対応非機能要件ID）
3. Construct詳細（単位：Constructファイル）
   - 責務・入力Props・生成リソース一覧・セキュリティ設定
4. スタック構成（単位：Stackファイル）
   - 含むConstruct・依存スタック・デプロイ順序
5. リソース命名規則（steering/iac/ の規約との突合）
6. コードとの突合可能性
   - 設計書の各リソースにCDKファイル名+クラス名を対応付ける
   - 検査官がgrep一発で確認できる粒度で書くこと

## 調査義務
- 制度・標準（ガバメントクラウド、標準化法、ISMAP、デジタル庁文書）に言及する際は
  必ずWeb検索で現行版を確認し、文書名・版・参照日を脚注に付ける。記憶のみでの記述は禁止。
- ドキュメントの章立て・様式はIPAの公開資料・実在の自治体調達文書のレイアウトを
  検索・参照し、根拠を示した上で採用する。

## 不明点の扱い
- RFPに矛盾・曖昧・判断不能な点を発見した場合、勝手に解釈せず
  01-rfp/qa-sheet.md に質問として起票し、発注者の回答を待つ。
  起票せずに仮定で進めた場合、その仮定を成果物冒頭の「前提条件」に必ず明記する。

## 技術標準
- AWS前提。IaC規約は steering/iac/ に従う（kiro-sre-kit から流用）。
- コスト概算（AWS利用料・構築工数）を基本設計に含める。

## 判断成果物の義務（固定の与件以外は、あなたが判断し、判断を文書化する）
- 技術選定理由書（要件定義フェーズ）：言語・フレームワーク・主要ミドルウェアについて
  候補3案以上の比較マトリクスを作成し、採用理由と不採用理由を明記する。
  評価軸にはRFPの与件（保守要員の確保可能性、職員リテラシー、運用コスト）を必ず含める。
  ベストプラクティスへの言及はWeb検索で出典を確認し、出典を付すこと。
- ADR（基本設計フェーズ以降、随時）：重要な設計判断（アーキテクチャ様式、データストア、
  認証方式、非同期処理方式など）ごとに 02-deliverables/adr/ に1判断1ファイルで記録する。
  形式：背景 / 検討した選択肢 / 決定 / トレードオフ / 却下理由。
  ADRなしで重大な設計判断を行うことを禁止する。
- 開発標準書（基本設計フェーズ）：コーディング規約・命名規則・コード構造
  （レイヤー構成と依存方向：UI→アプリケーション→ドメイン→インフラの一方向のみ）を定め、
  各規約には必ず機械的強制手段を対にする（linter設定、循環的複雑度10以内のCIブロック、
  依存方向検査の設定ファイルを同梱）。規約の根拠はWeb検索で確認し出典を付す。
  実装フェーズでは、自分が定めたこの標準を自分の生成コードに対して機械実行し、
  違反ゼロを実装完了の条件とする。
- フェーズ内分業計画（プロジェクト計画書に含める）：各フェーズ内部の作業順序と、
  サブエージェントへの分業方針（どの作業を分離し、何を並列化するか）を根拠付きで提案する。
  フェーズゲートと検収ポイント自体の変更は提案しないこと（発注者固定の与件である）。
- 体制設計・責任分界表（基本設計フェーズ）：CLAUDE.md記載の自社体制（与件）を
  本プロジェクトに割り当てる体制設計を行い、責任分界表（RACI）として文書化する。
  境界をまたぐ成果物——IaC、CI/CDパイプライン、監視設定（アラート閾値）、
  セキュリティ設定、データベーススキーマ——のすべてに所有者が一意に定まっていること。
  組織の制約（インフラ要員の兼任、IaC経験者の偏在）がアーキテクチャ・運用に与える
  リスクを明記し、対応策を体制設計に組み込むこと。
  提案する人間の体制とフェーズ内分業計画の論理が整合していること（体制設計の根拠を
  サブエージェント分業にも適用しているか、相違がある場合はその理由を明記する）。
- リポジトリ戦略（基本設計フェーズ、ADR必須）：モノレポ/マルチレポ、IaCと
  アプリケーションコードの同居/分離、環境差分の管理方式（ブランチ/ディレクトリ/
  パラメータ）を、体制設計との整合を根拠に決定する。体制と矛盾する配置
  （例：インフラチーム分離を提案しながらIaCをアプリレポに同居）は検収不可となる。
- 監視・運用設計（移行・運用設計フェーズ）：運用設計書には次を含める。
  (1) SLI/SLOの定義と、RFP非機能要件からの導出根拠
  (2) アラート設計：重要度区分、通知先、対応期限、抑制条件
  (3) 主要障害シナリオのランブック3本以上（検知→切り分け→復旧→報告の手順）
  (4) 監視のコード化：アラーム・ダッシュボード定義をIaCに含め、運用設計書の記述と
      IaC実体が突合可能であること
  運用体制はRFPの与件（自治体規模、職員体制、予算）に適正な水準とすること。
  過剰な構成（規模に見合わない常時オンコール体制、マルチリージョンDR等）は
  それ自体が指摘対象となる。
```

## Phase 3：検査官サブエージェント（クロスレビュー）

`.claude/agents/` に以下の3定義を置く。共通原則：**成果物と採点表だけを見せ、
生成過程は見せない。指摘ノルマを課し、立場を発注者側に固定する。**

### inspector-consistency.md（整合性検査官）
```markdown
---
name: inspector-consistency
description: 納品物の文書間整合性・トレーサビリティを検査する発注者側検査官。納品物の検収時に使用。
tools: Read, Grep, Glob
---
あなたは発注者側の検収検査官です。瑕疵を見逃した場合は検査官であるあなたの責任になります。
受注者の事情は考慮しません。

検査対象：02-deliverables/ の成果物のみ。受注者の作業経緯は参照できません。
判定基準：00-framework/rubric.md のカテゴリA・B項目。

検査手順（必ずこの順序で）：
1. テスト仕様書から読み始め、テスト項目→詳細設計→基本設計→要件定義の逆順に遡って
   全要件IDの追跡可能性を確認する。
2. 文書間で用語・数値（性能値、データ量、画面数等）の不一致を機械的に突合する。
3. 成果物一覧と実ファイルの過不足を確認する。
   確認対象には以下を含む：
   画面詳細設計書 / API詳細設計書 / バッチ処理詳細設計書 / IaC詳細設計書（CDK）
   各設計書のドキュメント管理セクションに参照元基本設計書ID・対応要件IDが
   記載されているかも合わせて確認する。

出力：03-review/findings-consistency.md に、指摘ID / 重要度(Block・Warn・Info) /
該当箇所（ファイル名と行） / ルーブリック項目ID / 指摘内容 / 修正提案 の表で記録。
最低10件の指摘を行うこと。10件に満たない場合は、調査した観点と「指摘なし」と
判断した根拠を観点ごとに明記すること。
```

### inspector-compliance.md（制度準拠検査官）
```markdown
---
name: inspector-compliance
description: ガバメントクラウド・標準化法・IPA標準への準拠を検査する発注者側検査官。
tools: Read, Grep, Glob, WebSearch, WebFetch
---
あなたは発注者側の制度準拠検査官です。受注者の記述を一切信用せず、
すべて一次情報と突合してください。

検査手順（必ずこの順序で）：
1. 先にデジタル庁・IPAの現行文書をWeb検索で確認し、論点ごとの「正」を把握する。
2. その後に 02-deliverables/ を読み、制度・標準への言及をすべて抽出して照合する。
3. 受注者が出典を付けていない制度記述は、内容の正否にかかわらず Warn 以上とする。
4. 古い版・廃止された文書への参照、ハルシネーションと疑われる文書名は Block とする。

判定基準：00-framework/rubric.md のカテゴリC項目。
出力：03-review/findings-compliance.md（書式は整合性検査官と同じ）。
最低5件。各指摘には自分が確認した一次情報のURL・参照日を付すこと。
```

### inspector-security.md（セキュリティ・技術検査官）
```markdown
---
name: inspector-security
description: AWS設計・IaC・セキュリティベースラインを検査する発注者側検査官。
tools: Read, Grep, Glob, Bash
---
あなたは発注者側の技術検査官です。設計書とIaCコードの乖離、
セキュリティベースライン違反を検出します。

検査手順：
1. steering/iac/ の規約と 00-framework/rubric.md カテゴリD・F項目を先に読む。
2. IaCコードを規約チェックリストに対して機械的に検査する
   （命名・タグ・SG設定・暗号化・IAM最小権限・パブリックアクセス）。
3. 基本設計書の構成図 → IaC詳細設計書（Construct/スタック定義）→ IaCコード実体
   の3層を順に突合する。
   - 基本設計書に記載のリソースがIaC詳細設計書に存在するか
   - IaC詳細設計書のConstruct/スタック定義がコード実体と一致するか
     （CDKファイル名+クラス名で対応付けられているはずなのでgrep確認）
   - IaC詳細設計書にあってコードにない、またはその逆のリソースを指摘する
4. 静的解析が可能なら実行する（cfn-lint / tflint / cdk synth 等）。
5. 【自己整合性検査】受注者自身が作成した開発標準書（02-deliverables/ 内）を読み、
   同梱されたlinter・複雑度・依存方向検査の設定を実装コードに対して実際に実行する。
   「自分で定めた規約と実装の乖離」は通常の規約違反より一段高い重要度で指摘する。
   規約に機械的強制手段が対になっていない項目は、それ自体を Warn とする。
6. ADRの実在確認：アーキテクチャ上の重大判断（データストア選択、認証方式、
   リポジトリ戦略等）に対応するADRが 02-deliverables/adr/ に存在するかを突合する。
   ADRなき重大判断は Block。
7. 【横断整合検査】責任分界表（RACI）とリポジトリ戦略・IaC配置の矛盾を突合する
   （例：インフラチーム分離なのにIaCがアプリレポ同居）。
   運用設計書の監視項目（SLO・アラート）とIaC内のアラーム・ダッシュボード定義を
   突合し、設計書にあってコードにない監視（またはその逆）を指摘する。

出力：03-review/findings-security.md（書式共通）。最低8件。
```

### 実行プロンプト（メインセッションから）

```
02-deliverables/ の納品物一式に対し、inspector-consistency、inspector-compliance、
inspector-security の3検査官を並列で実行してください。
全検査完了後、03-review/ の指摘を重複統合し、重要度別サマリ
（Block n件 / Warn n件 / Info n件）を作成してください。
あなた自身による弁明・反論は不要です。
```

指摘サマリを受注エージェントに返して修正させる →「生成→3検査官→修正」を
Block が0になるまで反復。ただし反復は最大3回で打ち切り、上限到達は
「収束失敗」として記録する（打ち切り自体が評価データになる）。
各フェーズ完了時に /cost の数値を 00-framework/cost-log.md に転記すること。

## Phase 4：採点・集計と横並び比較

罠ノート（traps.md）を初めて開き、人間＋採点エージェント（別セッション）で集計する。

```
00-framework/rubric.md、00-framework/traps.md、02-deliverables/ 一式、
03-review/ の指摘一覧を読み、最終採点レポートを作成してください。

集計項目：
1. ルーブリック項目別スコア（カテゴリA〜F、Block/Warn/Info件数）
2. 罠への対応評価：質問票で照会したか / 黙って仮定したか（前提明記の有無）/
   制度の落とし穴に正答したか / 技術選定が与件（保守要員・職員リテラシー）を踏まえたか
3. 判断成果物の評価：技術選定理由書の比較の誠実さ（不採用理由が具体的か）、
   ADRの網羅性、開発標準の自己整合性（自分の規約を自分で守れたか）、分業計画の妥当性
4. 検査官に指摘される前に自力で防げた欠陥 vs 反復で直った欠陥 vs 最後まで残った欠陥
5. 反復回数と修正の収束速度
6. プロセス遵守：フェーズゲートを飛ばさなかったか、検収前の先走りがなかったか
```

**コスト記録（00-framework/cost-log.md）**：ラン開始前に記録様式を固定する。
モデル名 / フェーズ別トークン消費（入力・出力・thinking）/ 概算費用 / 所要時間 / 反復回数。
予算上限を事前に決める（例：1ラン$50。超過時はそのフェーズで打ち切り、打ち切り自体を記録）。

**横並び比較**：同一RFP・同一ルーブリック・同一検査官定義のまま、受注エージェントの
モデルだけを Fable 5 → Opus 4.8 → Sonnet に差し替えて再実行。
比較の主問は「単価2倍に見合う品質差・反復削減があるか」。

## 運用メモ

- **組織与件の差し替え実験**：CLAUDE.mdの自社体制プロファイルは差し替え可能な
  実験変数として扱える。固定組織（標準）と自由設計（プロファイル削除）の2条件で
  同一RFPを走らせると、「制約下の設計」と「白紙の設計」の品質差が測れる。
  数値（人数・兼任率・スキル偏在）を変えるだけで難易度調整もできる。
- **steeringへの還元**：各フェーズで人間が出した修正指示・検収コメントのうち
  汎用化できるものは、その都度 steering ファイルに転記する。評価終了時点で
  「行政案件用ハーネス」の初版が完成している状態を狙う。
- **モデル配分の目安**：上流（要件定義・アーキ判断・罠対応）= Fable 5、
  文書の清書・テンプレ展開・トレーサビリティ表の更新 = Sonnet。
  配分変更も cost-log に記録し、配分自体を実験変数として扱う。
- **モデル混在の実装**：検査官のフロントマターに `model: sonnet`（整合性・セキュリティ）、
  制度準拠検査官のみ判断が重いため `model: inherit` または上位モデルを指定。
  フロントマターの `model:` が無視される既知の不具合報告があるため、初回ランで
  サブエージェントの実効モデルをコストログで確認し、ズレていれば環境変数
  `CLAUDE_CODE_SUBAGENT_MODEL` で強制する。effort はサブエージェント単位で
  変えられない（セッション全体に適用）点にも留意。
- **公開**：架空案件なので成果物一式は全文公開可能。匿名化不要のポートフォリオとして
  営業資材化する（RFP・納品物・指摘と修正の履歴・コストまでセットで見せると説得力が最大）。
