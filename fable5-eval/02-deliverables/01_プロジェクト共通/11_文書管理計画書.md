## 0. 本書の位置付けと提出遅延の経緯(お詫び)

1. 本書は、RFP第6章「記載粒度・様式は本市と協議の上、設計工程開始前に文書管理計画として確定する」およびプロジェクト計画書(KSM-PLN-001)§10「P1で文書管理計画書(記載粒度・様式・命名・改訂管理)を市と確定する」に基づく文書管理計画書である。
2. **本書はP1(要件定義)で確定すべきところ、提出が遅延し、P3時点での提出となった。** P1〜P2の間、命名・版数・改版履歴の運用は各成果物内で事実上統一されていたが(文書番号 KSM-xxx-nnn、版数、改版履歴表)、計画書としての文書化と市との確定を欠いていた。お詫びの上、G2検収コメント(ファイル名の日本語化)への対応方針と併せて本書で確定をお願いしたい。
3. 本書の確定(G3検収承認)をもって、§2の命名規則・§3〜§6の管理規則を全納品成果物に遡及適用する(遡及適用の実施結果は附表1)。

## 1. G2検収コメントへの対応方針(ファイル名の日本語化)

### 1.1 検収コメント

> 「ドキュメントのファイル名は、英語だと市の職員にはわかりにくい。日本語のファイル名のほうがわかりやすいので、そうしてもらえないか。」(G2検収結果・令和8年6月10日)

### 1.2 対応方針(決定)

| # | 方針 | 内容 |
|---|---|---|
| 1 | **納品文書のファイル名は日本語を正とする** | すべての納品文書(Markdown/PDF)のファイル名は、文書内表題と一致する日本語の正式名称とする(§2.1)。根拠:公文書管理の政府標準である内閣府「行政文書の管理に関するガイドライン」が、行政文書ファイルには内容を容易に理解できる分かりやすい名称を付すことを求めており[^1]、市職員が日常的に扱う文書管理の慣行に整合させる |
| 2 | **既存納品物(P0〜P2、検収済みを含む)も日本語名へ変更する** | 検収済み文書を含む全納品Markdown文書20件のファイル名を日本語へ変更した(附表1)。**ファイル名のみの変更であり、文書番号・版数・内容は一切変更しない**(§1.3) |
| 3 | **機械可読ファイルは例外として英語名を維持する** | ツールが規定名で探索する設定ファイル等(dev-standards/ 配下:checkstyle.xml、eslint.config.mjs、dependency-cruiser.cjs、ci-quality-gate.yml、.editorconfig、LayeredArchitectureTest.java)、ならびにP4以降に納品するソースコード・IaCコードのファイル名は、ツール仕様・コンパイラ制約により日本語化できないため英語名を維持する。これらは「文書」ではなく「コード成果物」であり、対応する説明文書(開発標準書・パラメータ設計書等)は日本語名で納品する |
| 4 | ~~ディレクトリ名はASCIIを維持する~~ **【1.1版で改定】納品文書フォルダは日本語名とする** | 1.0版の本方針(全ディレクトリASCII維持+納品媒体での読み替え一覧)は、G3検収コメントを受けて改定した。納品文書を格納する工程別フォルダは日本語名を正とし、機械可読資材(dev-standards/、将来のソースコード・IaC)のフォルダのみASCIIを維持する。判断理由・新構成は§7、命名規則は§2.4 |
| 5 | **相互参照の完全追従** | ファイル名変更時は、全文書の相互参照(ファイル名・パスの記載)を機械検索(grep)で抽出し追従させ、リンク切れ・参照齟齬ゼロを確認する(今回の実施結果:§1.4)。文書間参照は原則として**文書番号(KSM-xxx-nnn)で行い、物理ファイル名の直書きを避ける**ことを今後の規則とする |

### 1.3 既存納品物の扱い(版管理上の整理)

- ファイル名の変更は文書の記載内容に影響しないため、**改版(版数繰上げ)を伴わない構成管理上の措置**として扱う。変更の事実は本書附表1に記録し、構成管理台帳(§5)で追跡する。
- 検収済み文書(G0〜G2承認分)の内容は変更しない。ただし、旧ファイル名を本文中に直書きしていた2文書3箇所(附表2)は、参照齟齬(リンク切れ)を残さないため当該箇所のみ追従修正した。修正は「ファイル名表記の読み替え」に限定され、設計内容・要件・判断の変更を含まない。

### 1.4 実施結果(令和8年6月10日実施)

- 旧ファイル名の本文中参照3箇所を追従修正(附表2)。

## 2. 命名規則

### 2.1 納品文書(Markdown/PDF)

```
{工程フォルダ}/{連番2桁}_{日本語文書名}.md   例: 05_ソフトウェア方式設計・詳細設計/13_業務ルール詳細仕様書.md
adr/ADR-{連番3桁}_{日本語題名}.md            例: adr/ADR-011_帳票出力方式.md
分冊(1.2版追加):{連番2桁}_{親文書名}/{連番}-{枝番2桁}_{編名}.md
                                             例: 05_…/12_詳細設計書/12-03_抽選編.md
```

1. 日本語文書名は文書内表題(RFP第6章の成果物名称に対応)と一致させる。同一成果物が分冊となる場合は、**親文書名のサブフォルダを設け「親連番-枝番2桁_編名」を付す**(1.2版で規則化。例:12_詳細設計書/12-00_総説・共通編.md)。分冊の文書番号は親番号+枝番(例:KSM-DDD-001-03)とし、既存文書からの「KSM-DDD-001」参照は分冊群全体を指すものとして読み替える(読み替え表=分冊12-00 §1.3)。
2. 連番は納品順の通し番号とし、欠番・重複を禁止する(PMOが採番管理)。
3. ファイル名に版数は含めない。版数は文書内の管理情報欄・改版履歴で管理し、検収版はリポジトリのタグ・納品媒体で固定する(KSM-REP-001 §6と整合)。
4. ファイル名に使用できる文字:日本語(JIS X 0213範囲)、半角英数、アンダースコア、中黒(・)、ハイフン。機種依存文字・半角カナ・Windows予約文字(\\ / : * ? " < > |)を禁止する。

### 2.2 機械可読ファイル(例外)

dev-standards/ 配下の設定ファイル、P4以降のソースコード・IaCコード・CI定義は英語名(各ツール・言語の規定/慣行)とする。理由と対応文書は§1.2方針3のとおり。

### 2.4 フォルダ(ディレクトリ)【1.1版で追加】

```
{工程番号2桁}_{日本語工程名}/        例: 04_システム方式設計/
```

1. 納品文書フォルダは、RFP第6章「工程(共通フレーム対応)」の区分名に一致する日本語名とする(構成・根拠は§7)。工程番号はライフサイクル順の2桁とし、欠番・重複を禁止する(PMOが採番管理)。
2. フォルダ名の使用可能文字は§2.1第4項のファイル名規則と同一(JIS X 0213範囲の日本語+半角英数+アンダースコア・中黒・ハイフン。Unicode正規化はNFCに統一)。
3. 例外としてASCII名を維持するフォルダ:adr/(工程横断の設計判断記録。恒久指示〔CLAUDE.md〕により保存場所を 02-deliverables/adr/ に固定)、dev-standards/(機械可読資材)、P4以降に追加するソースコード・IaCコードの格納フォルダ。理由は§7.3。**【1.2版追加】ASCII名を維持するファイル:02-deliverables/module-index.md(発注者テンプレートが配置パスを指定する機械照合用の表=§8.2方針3)、kasumidai-yoyaku/docs/openapi.yaml(CI契約テストが参照する機械可読正本=§8.2方針4)。**
4. フォルダの新設・改名・廃止は本書の改版事項とし、PMOが管理する。各工程フォルダは当該工程の成果物初回納品時までに作成する(本版で全工程分を先行作成済み)。

## 3. 版数・改版管理

1. 版数は「X.Y」形式。X=検収承認を経た基線、Y=基線内の改版。初版は1.0。
2. すべての文書に管理情報欄(文書番号・版・作成日・作成者・承認状態)と改版履歴表(版・日付・改版内容・作成承認)を必須とする。
3. 検収承認された版は基線として固定し、以後の変更は改版(履歴記録+変更箇所の明示)による。検収済み文書の本文変更は、課題管理または検収指摘に紐づく場合のみ認める。
4. ファイル名のみの変更・誤字等の軽微訂正は版数を繰り上げず、本書附表または改版履歴の注記で記録する(§1.3)。

## 4. 記載粒度・様式

1. 章立て・様式はIPA「共通フレーム2013」のプロセス体系[^2]およびデジタル庁「デジタル社会推進標準ガイドライン(DS-100)」群[^3]の記載事項に基づく(P0〜P2成果物で適用済みの様式を踏襲)。
2. 制度・標準・ベストプラクティスへの言及は、Web検索で現行版を確認の上、文書名・版・URL・参照日を脚注に付す(CLAUDE.md調査義務。既存成果物で適用済み)。
3. 図はMermaid記法またはテキスト図とし、編集可能形式での納品(RFP第6章)を担保する。
4. 要件への言及は要件ID(REQ-xxx/NFR-xxx)で行い、トレーサビリティマトリクス(KSM-TRM-001)で追跡する。

└─ dev-standards/                        … 機械可読資材(linter設定・CI定義等。ASCII名維持=§7.3)

### 7.3 設計根拠

**(1) 工程区分の根拠:RFP第6章「工程(共通フレーム対応)」の区分をフォルダ区分として採用**

共通フレーム2013[^2]はプロセス・アクティビティ・タスクの体系(共通の物差し)であり、「フォルダ構成標準」を定めるものではない。そこで本計画では、**発注者・受注者間で既に合意済みの成果物体系である RFP第6章の表(工程=共通フレーム対応、の10区分)をそのままフォルダ区分として採用**した。これにより、(a)各文書の格納先がRFP第6章の納品成果物一覧と1:1に照合でき、(b)「どの工程の文書か」がフォルダ名で一目で分かる、という検収コメントの意図を満たす。各フォルダと共通フレーム2013のプロセス区分の対応は次のとおり。

| フォルダ | RFP第6章の工程区分 | 共通フレーム2013のプロセス区分(対応) |
|---|---|---|
| 01_プロジェクト共通 | プロジェクト共通 | プロジェクトプロセス(プロジェクト計画・アセスメント・統制)および支援プロセス(文書化・構成管理)。工程横断のため先頭に配置 |
| 02_企画プロセス | (RFP第6章に区分なし。当社追加成果物の区分) | 企画プロセス(システム化構想・システム化計画の立案)。RFP理解度確認書=P0企画の成果物 |
| 03_要件定義プロセス | 要件定義プロセス | 要件定義プロセス(利害関係者要件の定義) |
| 04_システム方式設計 | システム方式設計 | システム開発プロセスのシステム方式設計(全体構成・クラウド構成の決定) |
| 05_ソフトウェア方式設計・詳細設計 | ソフトウェア方式設計・詳細設計 | ソフトウェア実装プロセスのソフトウェア方式設計・ソフトウェア詳細設計 |
| 06_実装・テスト | 実装・テスト | ソフトウェア実装プロセスのソフトウェア構築・結合・適格性確認テスト(開発標準書・セキュア実装規約表は実装工程の作業標準としてここに区分) |
| 07_環境構築 | 環境構築 | システム開発プロセスの環境整備(IaCによる構築) |
| 08_移行 | 移行 | システム開発プロセスの導入(移行) |
| 09_運用準備 | 運用準備 | 運用プロセスの運用準備(運用テスト・利用者支援準備) |
| 10_検収・引渡し | 検収・引渡し | 合意プロセス(取得・供給)の検収・引渡し |
| 11_運用・保守 | 運用・保守(月次・年次) | 運用プロセス・保守プロセス |

**(2) 区分の原則:作成フェーズではなく「成果物が属する工程」で区分する**

例:開発標準書(KSM-DEV-001)・セキュア実装規約表(KSM-DEV-002)はP2〜P3で作成したが、実装工程の作業標準であるため 06_実装・テスト/ に区分した。テスト計画書(KSM-TSP-001)も同様(RFP第6章が「実装・テスト」区分にテスト計画書を置くことと整合)。トレーサビリティマトリクスは全工程で更新する台帳だが、RFP第6章が「要件対応表」を要件定義プロセスに区分しているため 03_要件定義プロセス/ に置く。

**(3) 工程横断フォルダの維持(adr/、dev-standards/)**

- **adr/**:ADRは基本設計以降「随時」追加される工程横断の設計判断記録であり、恒久指示(CLAUDE.md)が保存場所を 02-deliverables/adr/ に固定している(1判断1ファイル)。工程フォルダへ分散させると判断記録の通覧性(ADR-001〜011の単一コーパス)が失われるため、現行位置・ASCII名のまま維持する。
- **dev-standards/**:機械可読資材(linter設定・CI定義)。CIツールが参照するパスであり、§1.2方針3の例外区分として現行位置・ASCII名のまま維持する。

### 7.4 フォルダ名の日本語化の判断

| 対象 | 決定 | 理由 |
|---|---|---|
| 納品文書の工程別フォルダ(01〜11) | **日本語名を採用**(1.0版§1.2方針4を改定) | (a)検収コメントの要望に直接応える。(b)1.0版でASCII維持の根拠とした「CI・構成管理ツールとの互換性」を再評価した結果、納品文書(Markdown/PDF)はCIの機械処理対象ではなく、Git・Windows・主要CIともUTF-8パスを扱えるため、実害が想定されるのは機械可読資材と将来のコード成果物に限られる。(c)ファイル名(日本語)とフォルダ名(英語)の混在はかえって分かりにくい。リスク低減策として、文字集合の制限とNFC正規化(§2.4)、ソート安定化のためのASCII数字プレフィクスを併用する |
| adr/、dev-standards/、将来のソースコード・IaC格納フォルダ | **ASCII名を維持** | §7.3(3)のとおり、ツール参照パス・恒久指示・コンパイラ制約があるため。納品文書(人間が読む)と機械可読資材(ツールが読む)で扱いを分ける二層方式とする |

これに伴い、1.0版§1.2方針4の「納品媒体作成時の日本語フォルダ名読み替え一覧」は**不要となり廃止**する(リポジトリ構成と納品媒体構成が一致するため)。
