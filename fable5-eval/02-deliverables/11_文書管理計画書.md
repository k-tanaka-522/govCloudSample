## 0. 本書の位置付けと提出遅延の経緯(お詫び)

1. 本書は、RFP第6章「記載粒度・様式は本市と協議の上、設計工程開始前に文書管理計画として確定する」およびプロジェクト計画書(KSM-PLN-001)§10「P1で文書管理計画書(記載粒度・様式・命名・改訂管理)を市と確定する」に基づく文書管理計画書である。
2. **本書はP1(要件定義)で確定すべきところ、提出が遅延し、P3時点での提出となった。** P1〜P2の間、命名・版数・改版履歴の運用は各成果物内で事実上統一されていたが(文書番号 KSM-xxx-nnn、版数、改版履歴表)、計画書としての文書化と市との確定を欠いていた。お詫びの上、G2検収コメント(ファイル名の日本語化)への対応方針と併せて本書で確定をお願いしたい。
3. 本書の確定(G3検収承認)をもって、§2の命名規則・§3〜§6の管理規則を全納品成果物に遡及適用する(遡及適用の実施結果は附表1)。

## 1. G2検収コメントへの対応方針(ファイル名の日本語化)

### 1.1 検収コメント

> 「ドキュメントのファイル名は、英語だと市の職員にはわかりにくい。日本語のファイル名のほうがわかりやすいので、そうしてもらえないか。」(G2検収結果・令和8年6月10日)

### 1.2 対応方針(決定)

| # | 方針 | 内容 |
|---|---|---|
| 1 | **納品文書のファイル名は日本語を正とする** | すべての納品文書(Markdown/PDF)のファイル名は、文書内表題と一致する日本語の正式名称とする(§2.1)。根拠:公文書管理の政府標準である内閣府「行政文書の管理に関するガイドライン」が、行政文書ファイルには内容を容易に理解できる分かりやすい名称を付すことを求めており[^1]、市職員が日常的に扱う文書管理の慣行に整合させる |
| 2 | **既存納品物(P0〜P2、検収済みを含む)も日本語名へ変更する** | 検収済み文書を含む全納品Markdown文書20件のファイル名を日本語へ変更した(附表1)。**ファイル名のみの変更であり、文書番号・版数・内容は一切変更しない**(§1.3) |
| 3 | **機械可読ファイルは例外として英語名を維持する** | ツールが規定名で探索する設定ファイル等(dev-standards/ 配下:checkstyle.xml、eslint.config.mjs、dependency-cruiser.cjs、ci-quality-gate.yml、.editorconfig、LayeredArchitectureTest.java)、ならびにP4以降に納品するソースコード・IaCコードのファイル名は、ツール仕様・コンパイラ制約により日本語化できないため英語名を維持する。これらは「文書」ではなく「コード成果物」であり、対応する説明文書(開発標準書・パラメータ設計書等)は日本語名で納品する |
| 4 | **ディレクトリ名はASCIIを維持する** | 02-deliverables/、adr/、dev-standards/ 等のディレクトリ名は、CI・構成管理ツールとの互換性確保のため半角英数を維持する。市納品媒体(電子媒体)作成時は、日本語フォルダ名(「02_納品成果物」等)への読み替え一覧を添付する |
| 5 | **相互参照の完全追従** | ファイル名変更時は、全文書の相互参照(ファイル名・パスの記載)を機械検索(grep)で抽出し追従させ、リンク切れ・参照齟齬ゼロを確認する(今回の実施結果:§1.4)。文書間参照は原則として**文書番号(KSM-xxx-nnn)で行い、物理ファイル名の直書きを避ける**ことを今後の規則とする |

### 1.3 既存納品物の扱い(版管理上の整理)

- ファイル名の変更は文書の記載内容に影響しないため、**改版(版数繰上げ)を伴わない構成管理上の措置**として扱う。変更の事実は本書附表1に記録し、構成管理台帳(§5)で追跡する。
- 検収済み文書(G0〜G2承認分)の内容は変更しない。ただし、旧ファイル名を本文中に直書きしていた2文書3箇所(附表2)は、参照齟齬(リンク切れ)を残さないため当該箇所のみ追従修正した。修正は「ファイル名表記の読み替え」に限定され、設計内容・要件・判断の変更を含まない。

### 1.4 実施結果(令和8年6月10日実施)

- 納品Markdown文書20件(本編10+ADR10)を日本語ファイル名へ変更(附表1)。
- 旧ファイル名の本文中参照3箇所を追従修正(附表2)。
- 機械検索により、旧英語ファイル名への残存参照が0件であることを確認した(検索パターン:旧ファイル名20件の完全一致)。

## 2. 命名規則

### 2.1 納品文書(Markdown/PDF)

```
{連番2桁}_{日本語文書名}.md         例: 12_詳細設計書.md
adr/ADR-{連番3桁}_{日本語題名}.md    例: adr/ADR-011_帳票出力方式.md
```

1. 日本語文書名は文書内表題(RFP第6章の成果物名称に対応)と一致させる。同一成果物が分冊となる場合は「_編名」を付す(例:07_基本設計書_システム方式設計編.md)。
2. 連番は納品順の通し番号とし、欠番・重複を禁止する(PMOが採番管理)。
3. ファイル名に版数は含めない。版数は文書内の管理情報欄・改版履歴で管理し、検収版はリポジトリのタグ・納品媒体で固定する(KSM-REP-001 §6と整合)。
4. ファイル名に使用できる文字:日本語(JIS X 0213範囲)、半角英数、アンダースコア、中黒(・)、ハイフン。機種依存文字・半角カナ・Windows予約文字(\\ / : * ? " < > |)を禁止する。

### 2.2 機械可読ファイル(例外)

dev-standards/ 配下の設定ファイル、P4以降のソースコード・IaCコード・CI定義は英語名(各ツール・言語の規定/慣行)とする。理由と対応文書は§1.2方針3のとおり。

## 3. 版数・改版管理

1. 版数は「X.Y」形式。X=検収承認を経た基線、Y=基線内の改版。初版は1.0。
2. すべての文書に管理情報欄(文書番号・版・作成日・作成者・承認状態)と改版履歴表(版・日付・改版内容・作成承認)を必須とする。
3. 検収承認された版は基線として固定し、以後の変更は改版(履歴記録+変更箇所の明示)による。検収済み文書の本文変更は、課題管理または検収指摘に紐づく場合のみ認める。
4. ファイル名のみの変更・誤字等の軽微訂正は版数を繰り上げず、本書附表または改版履歴の注記で記録する(§1.3)。

## 4. 記載粒度・様式

1. 章立て・様式はIPA「共通フレーム2013」のプロセス体系[^2]およびデジタル庁「デジタル社会推進標準ガイドライン(DS-100)」群[^3]の記載事項に基づく(P0〜P2成果物で適用済みの様式を踏襲)。
2. 制度・標準・ベストプラクティスへの言及は、Web検索で現行版を確認の上、文書名・版・URL・参照日を脚注に付す(CLAUDE.md調査義務。既存成果物で適用済み)。
3. 図はMermaid記法またはテキスト図とし、編集可能形式での納品(RFP第6章)を担保する。
4. 要件への言及は要件ID(REQ-xxx/NFR-xxx)で行い、トレーサビリティマトリクス(KSM-TRM-001)で追跡する。

## 5. 構成管理・納品

| 項目 | 規則 |
|---|---|
| 保管場所 | 02-deliverables/ 配下で一元管理(KSM-PLN-001 §10) |
| 構成管理 | リポジトリで版管理し、検収版にタグを付す(KSM-REP-001 §6)。納品成果物一覧(連番・文書番号・版・検収状態)はPMOが台帳管理し、各フェーズ検収資料に添付する |
| 納品媒体 | 電子媒体:PDF+編集可能形式(Markdown)(RFP第6章)。媒体内フォルダ名の日本語読み替え一覧を添付(§1.2方針4) |
| 検収フロー | PMO機械検査(要件ID網羅・文書間突合・命名規則)→内部レビュー→検収依頼→市承認(KSM-PLN-001 §7)。市側の書面決裁は5営業日を標準とする(リスクR-08整合) |

## 6. 本フェーズ(P3)での適用確認

### 6.1 P3成果物の命名適用

P3成果物(12〜15、ADR-011、TRM 1.2版)はすべて§2.1の日本語命名規則で作成した(G3検収依頼書の成果物一覧参照)。

### 6.2 KSM-BDD-002(基本設計書アプリ編)の欠番処理

G2残課題3に対する市回答(「P3詳細設計と内容が重複するため、P3成果物に統合して構わない。画面・帳票・DB・外部IFの設計はP3の詳細設計書で提出」)に基づき、KSM-BDD-002は**発行せず欠番**とし、その設計範囲は詳細設計書(KSM-DDD-001)に統合した。KSM-BDD-001 §12残課題3およびKSM-TRM-001の「BDD-002(R9.1追補)」参照は、KSM-TRM-001 1.2版でKSM-DDD-001の該当章へ読み替え済み。
