# KSM-ADR-005 データストア(P1方針の正式化)

| 項目 | 内容 |
|---|---|
| ステータス | 承認待ち(G2) |
| 日付 | 令和8年6月10日 |
| 起案 | リードA(アーキテクト) |
| 関連 | REQ-005、NFR-A02〜A04、NFR-B02、NFR-D02、NFR-E01、KSM-TEC-001 §5(P1比較)、KSM-BDD-001 §3/§5 |

## 背景

REQ-005(全データをガバメントクラウド上のマネージドDBで一元管理。QA No.1でNFR-E03削除が確定)に基づくデータストアを正式決定する。P1の技術選定理由書(KSM-TEC-001 §5)で4案比較(RDS for PostgreSQL/Aurora PostgreSQL/RDS for MySQL/DynamoDB)を実施済みであり、本ADRはその決定の正式化と設計値の確定を行う。

業務特性:予約・料金・減免・収納はACIDトランザクション(一括予約の全件成立/不成立=REQ-010)と多軸集計(統計帳票・年度集計=REQ-025)が中核。データ規模は小さい(利用者18,000件(5年後)、予約5年累計約30万件)。

## 検討した選択肢

KSM-TEC-001 §5.1の比較マトリクスを正とする(再掲省略)。要点:

- **A案 RDS for PostgreSQL(マルチAZ)**:11点。業務特性適合・地域内保守要員(PostgreSQLはOSS RDBの事実上の標準)・本規模で最安。
- **B案 Aurora PostgreSQL**:8点。上位性能が本規模では過剰でコスト劣位(最小構成でもRDS比約2倍)。
- **C案 RDS for MySQL**:7点。同等だが新規開発の主流性・積極採用理由でPostgreSQLに劣後。
- **D案 DynamoDB**:3点。ACID・多軸集計の実装複雑化、NoSQL設計スキルの地域内確保困難。

## 決定

**Amazon RDS for PostgreSQL(マルチAZ配置)を正式採用**し、設計値を次のとおり確定する。

| 項目 | 設計値 |
|---|---|
| インスタンス | db.t4g.medium(2vCPU/4GB)マルチAZ。検証はdb.t4g.smallシングルAZ |
| エンジン | PostgreSQL 現行メジャー版(P4構築時の最新安定版を採用し、サポート期限をパラメータ設計書に記録) |
| ストレージ | gp3 100GB(自動拡張上限200GB。NFR-B02の2倍余裕) |
| バックアップ | 自動バックアップ日次・保持7日(7世代)+PITR(NFR-A03/A04) |
| 暗号化 | KMSカスタマー管理キー(KSM-ADR-010)(NFR-E01) |
| 文字コード | UTF-8(JIS X 0213範囲対応・外字不使用=REQ-027) |
| 接続 | 私設サブネット内のみ。アプリからはSecrets Manager経由の認証情報 |
| 移行性 | スキーマ・データはレイアウト定義書付きでエクスポート可能(pg_dump/CSV)とし、NFR-D02(市帰属・ロックイン回避)を担保 |

## トレードオフ

- マルチAZはシングルAZの約2倍の単価(本番月約200ドル)→ 稼働率99.5%・自動フェイルオーバー(NFR-A02)の要件上必要であり許容。検証環境はシングルAZ+停止運用でコスト最小化。
- t4g(バースト型)はベースライン超過の持続負荷に弱い → 本件の定常負荷はベースライン内であり、月次ピークは短時間。P5性能テストでCPUクレジット消費を実測し、不足時はm7g系へのスケールアップで対処(インスタンス変更のみ。アーキテクチャ変更なし=NFR-B02)。

## 却下理由

- **Aurora PostgreSQL**:本規模では性能余力を使う場面がなくコストのみ増加(設計方針D-5:適正水準)。利用者数が想定を大きく超えた場合はスナップショット移行が容易であり、移行パスとして温存。
- **RDS for MySQL**:機能同等だが、PostgreSQLを上回る積極理由がなく、新規開発の主流性・地域内技術者層でPostgreSQL優位(KSM-TEC-001 §5.2)。
- **DynamoDB**:一括予約のACIDトランザクション・統計の多軸集計が実装複雑化し工数増。NoSQL設計スキルの地域内確保(NFR-C05)も困難。
