## 背景(Context)

KSM-DDD-001 §6.1 はバッチジョブ(JB-01 抽選・JB-02 保留解放・JB-03 決済期限・JB-04 日次統計)の実行方式として「バッチジョブランナー」の実装を規定している。KSM-ADR-008 は「バッチをアプリケーションサーバに組み込み、EventBridge Scheduler で ECS RunTask として起動する」方針を定めているが、具体的なバッチフレームワーク(Spring Batch の採否等)については明示的な決定記録がなかった。

P4 実装フェーズにおいて、Spring Batch の採用可否を評価した結果、採用しないことを決定した。本 ADR はその決定理由・代替案の評価・トレードオフ・否定理由を誠実に記録するものであり、KSM-IMP-001 1.1版 §4.1 の乖離 D-2 に対応する文書上の補完である。

---

## 評価した選択肢

### 選択肢 A: Spring Batch 採用

**概要**: Spring Batch の `Job` / `Step` / `JobRepository` を使用し、ジョブ定義・実行状態をジョブリポジトリ(RDS テーブル)で管理する。

**利点**:
- ジョブ実行状態(成功/失敗/再実行可能性)がJobRepositoryで一元管理
- チャンクオリエンテッド処理(大量データのバッチ処理)に最適化
- Spring エコシステムとの統合が容易
- 再実行・スキップ・リトライの宣言的設定

**欠点**:
- ECS Fargate でのステートレス実行と Spring Batch のステートフルな JobRepository が設計方針上の摩擦を生む
- JobRepository 用の追加テーブル(BATCH_JOB_INSTANCE 等 9テーブル)が RDS に必要
- コールドスタート時(ECS RunTask)にフレームワーク初期化コストが増大(抽選 JB-01 は月1回のため顕著)
- 抽選・保留解放等の業務ロジックはすでにドメイン層に実装済みであり、Spring Batch の ItemReader/ItemProcessor/ItemWriter へのマッピングが過剰な抽象化となる
- 運用チーム(0.5人月/月)が Spring Batch 固有の運用知識(JobExplorer・JobOperator・ジョブパラメータ等)を習得する必要がある

### 選択肢 B: ApplicationRunner + batch_job_locks テーブル(採用)

**概要**: `ApplicationRunner` を実装した `BatchJobRunner` を作成し、`--job=JB-xx` の起動引数でジョブを切り替える。冪等性と多重起動防止は `batch_job_locks` テーブル(アプリケーション層のロックテーブル)で担保する。

**利点**:
- Spring Batch の追加テーブル・設定・学習コストが不要
- ECS RunTask のコールドスタート・ステートレス実行と完全に適合
- 業務ロジックはドメイン層(`LotteryExecutionService`・`HoldReleaseService`等)に既実装済みであり、`BatchJobRunner` は呼び出し口に徹する最小構成
- `batch_job_locks` テーブルによる冪等性担保はシンプルで透明性が高い(アプリロジックで完結)
- 運用チームが習得すべき概念が少ない(Spring Batch 固有の操作不要)

**欠点**:
- 大量データのチャンク処理(将来的に数万件規模)での最適化は自前実装が必要
- ジョブ実行履歴の可視化は Spring Batch の JobRepository ほど網羅的でない(CloudWatch Logs + カスタムメトリクスで代替)
- 再実行ロジック(部分失敗時のリジューム)を自前で実装する場合は追加工数が発生

### 選択肢 C: AWS Batch 採用

**概要**: AWS Batch サービスを使用してジョブキューとコンピュート環境を定義する。

**評価**: 月1回・日1回程度の軽量バッチに AWS Batch のオーバーヘッド(コンピュート環境・ジョブキュー・ジョブ定義の管理)は過剰。ECS Fargate との統合も ADR-001 の方針(ECS 単一コンピュートプレーン)から逸脱する。**不採用**。

---

## 決定(Decision)

**選択肢 B: ApplicationRunner + batch_job_locks テーブルを採用する。**

理由:
1. **KSM-ADR-008 との整合**: ADR-008 が定める「ECS RunTask でバッチを起動」方針と最も適合する。ECS RunTask はステートレスなコンテナ起動であり、Spring Batch の ステートフルな JobRepository との整合性に課題が生じる。
2. **業務ロジックの再利用**: 抽選・保留・決済期限・統計の各ビジネスロジックはドメイン層に完成済み。Spring Batch の ItemProcessor 抽象化を追加する必要がなく、呼び出しコードのみを `BatchJobRunner` に実装するだけで足りる。
3. **運用体制への適合**: 0.5人月/月の運用体制(CLAUDE.md 記載)では、Spring Batch 固有の運用操作(ジョブ再実行コマンド・JobExplorer 操作等)の習得コストが運用リスクとなる。シンプルな構成が保守性に優る。
4. **冪等性の担保**: `batch_job_locks` テーブルによる SELECT FOR UPDATE ベースのロック取得は、ECS Fargate の多重起動抑制として実証された手法であり、Spring Batch の JobRepository と等価の多重起動防止機能を提供する。

---

## 結果・トレードオフ(Consequences)

**許容するトレードオフ**:
- ジョブ実行履歴は Spring Batch の JobRepository ほど詳細でない。CloudWatch Logs のジョブ開始/終了ログ + OPS-ALM-012(バッチ JB-01 失敗アラーム)で代替する。
- 将来的に数万件規模のチャンク処理が必要になった場合は、Spring Batch または Amazon EMR への移行を再評価する。現在の処理規模(抽選 2,000〜2,500 件・保留解放は件数次第)は ApplicationRunner で対応可能と判断した。

**Spring Batch 非採用の明示的否定理由**:
- ECS RunTask(コールドスタート)× Spring Batch(JobRepository 初期化・データソース接続)の組み合わせは、月1回の抽選ジョブで許容できないレイテンシリスクがある
- JobRepository 用 9テーブルの追加は、ステートレス設計(ADR-006 IaC 管理)と相反する永続状態の増加を招く
- 運用チームの技術スタック(Spring Boot の一般知識)の範囲内で完結させる判断を優先した

**今後の対応**:
- KSM-DDD-001 §6.1 の記述を実態(ApplicationRunner + batch_job_locks)に合わせて訂正する(P5 前に実施)
- KSM-TRM-001 §4.3 D-2 の乖離記録で本 ADR を参照する
