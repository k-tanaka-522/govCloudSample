# KSM-ADR-011 帳票出力方式(PDF生成)

| 項目 | 内容 |
|---|---|
| ステータス | 承認待ち(G3) |
| 日付 | 令和8年6月10日 |
| 起案 | リードB(業務チームB)/リードA承認 |
| 関連 | REQ-017(納付書)、REQ-025(統計帳票CSV/PDF)、REQ-018/019、NFR-C05(地域内保守要員)、KSM-DDD-001 §2(帳票設計)、KSM-ADR-001(実行基盤) |

## 背景

帳票9種(KSM-DDD-001 §2.1)のうちPDF出力7種の生成方式を決定する。要件:(1)納付書(RP-02)はコンビニ収納用GS1-128バーコード・固定様式の精密なレイアウトが必須(REQ-017)。(2)統計帳票(RP-04/05)は表形式・ページ繰り・年度集計(REQ-025)。(3)生成はECS Fargate上のJavaアプリ内で完結させたい(実行基盤の追加はKSM-ADR-001の単一基盤方針に反する)。(4)保守はJava要員で可能であること(NFR-C05)。(5)追加ライセンス費は運用保守費(月額600千円)を圧迫しないこと。

P2では方式未決のまま帳票設計をアプリ編(→P3統合)へ送っており、本決定は詳細設計上の新規判断であるためADRとして記録する。

## 決定

**A案:JasperReports Library を採用**する。

1. PDF帳票(RP-01〜07)はバックエンド(Spring Boot)内のJasperReportsで生成し、S3(帳票バケット・CMK暗号化)へ保存の上、署名付き一時URLでダウンロード提供する(個人情報を含む帳票のキャッシュ禁止=Cache-Control: no-store)。
2. 納付書(RP-02)のGS1-128バーコードはJasperReports標準のバーコードコンポーネント(barcode4j)で生成し、GS1 Japanの標準料金代理収納ガイドライン準拠[^2]を結合テストで実スキャナ検証する。
3. 帳票定義(jrxml)はリポジトリで版管理し(様式変更=PR+レビュー)、様式IDと版を帳票フッタに印字して監査可能とする。
4. 大量出力(年度集計等)は非同期生成(SQS→ワーカー)とし、API応答をブロックしない(KSM-ADR-008と整合)。

## トレードオフ

- jrxml帳票定義の学習が必要 → 当社標準部品(過去案件の帳票枠組み)を再利用し、帳票デザインはリードB配下で集約(KSM-ORG-001:収納・職員系のA=リードB)。
- ライブラリ依存(脆弱性追従)が増える → 依存脆弱性検査(KSM-DEV-002 S-31)の対象として機械管理。直近もCVE対処が継続供給されており(2026年3月リリースで対処実績[^1])、保守継続性を確認済み。
- LGPLライブラリの利用 → ライブラリとしての利用(改変なし・動的リンク相当)であり、成果物の市帰属(RFP第8章。汎用OSS部分は対象外)と両立する。license-checker許容リスト(KSM-DEV-002 S-33)に登録。

## 却下理由

- **HTML→PDF変換(B案)**:画面系の簡易PDFには適するが、納付書の固定桁割・バーコード精度・段組の制御を自作する工数がJasperReports採用を上回る。帳票枠組みの自作は保守対象を増やすだけで利点がない。
- **帳票SaaS/外部帳票サーバ(C案)**:個人情報(氏名・住所)を外部サービスへ送信する整理(ISMAP・市セキュリティポリシー適合確認)と継続ライセンス費が発生し、本件規模に過剰(適正水準の原則=KSM-BDD-001 D-5)。
- **ブラウザ印刷(D案)**:公金収納に用いる納付書・議会監査に供する統計帳票は再現性のある固定様式が必須であり、ブラウザ・プリンタ依存の印刷では品質を保証できない。画面の簡易印刷(参考出力)としては併用可。
