## 1. プロジェクト概要

### 1.1 目的

人口約5万人の霞台市における計34公共施設の予約管理システムを、ガバメントクラウド(AWS)上に IaC により構築し、令和10年1月4日に本番稼働させ、以後5年間の運用・保守を行う。現行オンプレミス(保守期限・パッケージ終息)からの脱却、運用経費(5年総額)2割以上削減、スマートフォン最適化・キャッシュレス決済による市民サービス向上を実現する。

### 1.2 スコープ

RFP 第1章 1.4 のとおり:(1)設計・開発 (2)ガバメントクラウド上の本番・検証環境構築(IaC) (3)データ移行(利用者約12,000件・施設マスタ・将来予約約8,000件) (4)教育・研修 (5)稼働後5年間の運用・保守。クラウド利用料は市負担、受注者はクラウド事業者への技術的対応を行う。

### 1.3 固定与件(変更不可)

1. ガバメントクラウド(AWS)国内リージョン
2. IPA「共通フレーム2013」準拠のプロセス・成果物体系
3. 総務省セキュリティガイドライン・市ポリシー適合、ISMAP 登録サービスの利用を基本(AWS は ISMAP 登録済みであることを確認した[^4])
4. 標準仕様書適合性確認(対象文書は Q2 照会中)
5. 本番稼働日:令和10年1月4日(変更不可)
6. フェーズゲートと発注者検収ポイント(変更提案しない)

## 2. 適用プロセス標準とフェーズゲート

IPA「共通フレーム2013」[^2]に基づくフェーズゲート方式を適用する。各フェーズの成果物を 02-deliverables/ に保存し、発注者検収の承認をもって次フェーズへ着手する(検収前の先行着手禁止)。

```
企画(P0) → 要件定義(P1) → 基本設計(P2) → 詳細設計(P3) → 実装(P4) → テスト(P5) → 移行・運用設計(P6) → 本番稼働 → 運用・保守
   ▲G0        ▲G1            ▲G2            ▲G3           ▲G4         ▲G5          ▲G6(移行判定)
```

全要件に要件ID(REQ-001〜028、NFR-A01〜F02)を付番済み(RFP 付番を踏襲)し、要件定義書→設計書→テスト仕様書→IaC まで追跡するトレーサビリティマトリクスを P1 で初版作成し、以後全フェーズで更新する。

## 5. 実施体制

### 5.1 与件(自社体制:変更不可)

- アプリケーション開発部:8名(Java/TypeScript 中心。AWS 経験は半数、IaC 経験者は1名)
- インフラ基盤部:3名(全員兼任。本件充当は合計 0.5人月/月)
- PMO:1名(品質・進捗管理。技術判断は行わない)
- 納品後の保守運用:自社運用チーム(夜間休日は外部委託の一次受付のみ)

### 5.2 プロジェクト体制図

```
発注者(霞台市 情報政策課)
   │ フェーズ検収・課題協議(週次定例/月次工程会議)
   ▼
PM(アプリ開発部シニア・1名)──── PMO(1名:品質・進捗・トレーサビリティ管理)
   │
   ├─ 業務チームA「予約・利用者」(リードA+2名=3名)
   │     利用者管理(REQ-001〜005)、空き照会・予約・抽選(REQ-006〜014)、通知
   │     ※リードAはアーキテクト兼務(技術選定・開発標準・ADR 起案)
   │
   ├─ 業務チームB「料金収納・職員管理・移行」(リードB+2名=3名)
   │     料金・収納・減免・還付(REQ-015〜020)、職員機能(REQ-021〜026)、
   │     外部IF(決済代行・財務会計CSV)、データ移行(REQ-028)
   │
   ├─ 基盤チーム「クラウド・IaC・CI/CD」(IaC経験者1名+TS開発者1名=2名)
   │     AWS構成、IaC、CI/CD、監視、性能テスト、環境構築
   │     ◄─ インフラ基盤部(0.5人月/月):設計レビュー、ガバクラ接続・
   │         セキュリティ専門点検に限定投入(実作業は基盤チームが実施)
   │
   └─ 運用チーム(納品後):運用・保守、ヘルプデスク(開庁日8:30〜17:15)
         夜間休日は外部委託の一次受付のみ → アラート自動通知+翌開庁日対応を
         基本とする運用設計(NFR-C02 の1時間以内第一報との整合は P6 で設計)
```

アプリ開発部8名の配置:PM 1名+業務チームA 3名+業務チームB 3名+基盤チーム1名(TS開発者。IaC経験者1名は部内のIaC経験者を充当)= 8名。フェーズにより重心を移動する(P1 は業務チーム重心、P4〜P5 は全員実装・テスト、P6 は移行・運用重心)。

### 5.3 組織制約と技術判断への影響・対応策

| 制約 | 影響(緊張関係) | 対応策 |
|---|---|---|
| インフラ基盤部は 0.5人月/月のみ(専任ゼロ) | インフラ専任チームを置く体制は組めない。IaC・環境構築をアプリ部が担うしかない | (1) IaC はアプリ部の TypeScript スキルを活かせるツール(AWS CDK(TypeScript)を有力候補。正式決定は技術選定理由書・ADR による)を優先検討し、アプリ部内製とする。(2) インフラ基盤部はレビューゲート(方式設計・セキュリティ・ガバクラ手続)に限定投入し 0.5人月/月に収める。(3) 環境構築・変更は CI/CD パイプラインで完全自動化(セルフサービス化)し、手作業構成変更を禁止(NFR-C01 と整合) |
| AWS 経験者がアプリ部の半数、IaC 経験者1名 | IaC・クラウド設計が単一障害点(属人化) | (1) P1〜P2 期間中に AWS 研修(認定資格レベル)を業務チームリード2名へ実施。(2) IaC はペア作業(経験者+TS開発者)で実施しナレッジ移転。(3) steering/iac 規約+静的検査(cdk-nag 等)で品質を機械的に担保し、レビュー負荷を下げる |
| PMO は技術判断を行わない | 技術判断の責任者が必要 | 技術判断はリードA(アーキテクト)に一元化し、ADR で文書化。PMO は ADR・トレーサビリティ・規約検査の「存在と整合」を機械的に検査する役割分担 |
| 運用チームの夜間休日は外部委託一次受付のみ | 24時間365日サービス(NFR-A01)と重大障害1時間以内第一報(NFR-C02)への対応 | マネージドサービス中心の構成で夜間の人手対応を最小化。アラートの自動通知・自動復旧(Auto Scaling、マネージドDBフェイルオーバー)を設計の前提とし、外部委託一次受付のエスカレーション基準を P6 で文書化。市の規模(人口5万人)に対し常時オンコール体制は過剰であり採らない |

詳細な責任分界表(RACI:IaC・CI/CD・監視設定・セキュリティ設定・DBスキーマの所有者一意化)は、CLAUDE.md の定めに従い P2(基本設計)で「体制設計・責任分界表」として文書化する。

### 5.4 発注者側体制(別紙3より)

事務局(情報政策課係長・主事、充当5割)が窓口。業務担当(スポーツ振興課2・生涯学習課2・文化振興課1)、会計課1名は兼務であるため、要件確認・受入テストの市側作業は「週2回・半日単位」の定例枠に集約して依頼する(兼務職員の負荷平準化)。

## 6. フェーズ内分業計画

各フェーズ内部の作業順序と、サブエージェント(AI 作業単位)への分業方針を定める。フェーズゲートと検収ポイント自体は変更しない(発注者固定の与件)。

### 6.1 分業の設計原則(根拠)

1. **人間体制との同型性**:サブエージェントの分割は 5.2 の人間チーム分割(業務ドメイン2分割+基盤横断)と同じ境界で行う。理由:成果物の所有者(人間)とレビュー責任が1:1で対応し、統合時の責任不明領域が生じないため。人間体制とサブエージェント分業で境界が異なる箇所は本計画に明記する(現時点では同一)。
2. **書く者と検べる者の分離**:生成(設計・コード)エージェントと検証(規約検査・突合・トレーサビリティ更新)エージェントを分離する。理由:自己検証の見落とし防止。PMO の役割(品質の機械検査)と同型。
3. **統合点は直列・単一**:横断成果物(用語集、DB スキーマ、トレーサビリティマトリクス、ADR 採番)は単一エージェント(リードA 相当)が直列に統合する。理由:並列更新による不整合(同名異義・採番衝突)の防止。
4. **並列化は「相互依存が要件IDレベルで分離可能な単位」に限る**:REQ の機能群(利用者管理/予約・抽選/料金収納/職員管理)は相互参照が少なく並列化可能。DB スキーマ・共通部品は依存の根であるため先行直列。

### 6.2 フェーズ別の作業順序と分業

| フェーズ | 直列の背骨(順序固定) | 並列化する作業(サブエージェント分業) | 統合・検証 |
|---|---|---|---|
| P1 要件定義 | 用語集・業務全体フロー → 機能群別詳細化 → 要件対応表統合 | (a)利用者管理+予約・抽選(REQ-001〜014)、(b)料金収納+職員管理(REQ-015〜026)、(c)非機能(NFR 全項)、(d)技術選定の候補調査(3案比較の情報収集) の4並列 | 統合エージェントが要件対応表・用語整合を直列統合 → 検証エージェントが REQ/NFR 網羅性を機械突合 → 人間リードがレビュー |
| P2 基本設計 | システム方式・DB 論理設計(先行直列) → 画面・帳票・外部IF 設計 | (a)画面設計(利用者系/職員系で2並列)、(b)帳票・CSV 外部IF、(c)クラウド方式・監視方式、(d)開発標準書+linter 設定 | ADR は1判断1ファイルでリードA 相当が直列起案。構成図⇔IaC 対応表の雛形を同時作成 |
| P3 詳細設計 | DB 物理・共通部品詳細(先行) → モジュール別詳細設計 | 機能群別(A/B チーム境界)+IaC 詳細+テスト設計 の並列 | テスト設計は生成と別エージェントが要件ID 起点で作成(検証分離) |
| P4 実装 | 共通基盤・スキーマ・CI/CD(先行) → 機能群別実装 | 機能群別実装を要件ID 単位で並列。IaC 実装は単一エージェント直列(規約逸脱・リソース衝突防止) | CI で開発標準を機械実行(複雑度・依存方向・lint)。違反ゼロが完了条件 |
| P5 テスト | 結合 → 総合 → 性能 → 脆弱性対処 | テスト実施ログ整理・報告書化を機能群別に並列 | 性能テストはピークシナリオ(抽選初日)を基盤担当が専任 |
| P6 移行・運用設計 | 移行設計 → リハ1 → 是正 → リハ2 → 判定 | (a)移行ツール・データ検証、(b)運用設計書・ランブック、(c)マニュアル・研修資料 の3並列 | 監視項目一覧⇔IaC アラーム定義の機械突合(steering/iac 規約 5.) |

### 6.3 人間体制との整合確認

- 並列単位(a)(b)は業務チームA/B の担当範囲と一致、(c)(d)系は基盤チーム/リードA の担当範囲と一致する(原則1)。
- IaC 実装を「単一エージェント直列」とするのは、人間側で IaC 経験者が1名であり、レビュー可能な変更量に絞る必要があるためであり、体制設計の根拠(レビューボトルネックの保護)をそのまま分業に適用したものである。
- 相違点:現時点でなし。フェーズ着手時の詳細化で相違が生じた場合は理由を当該フェーズ計画に明記する。

## 7. 品質管理・構成管理

1. **フェーズゲート検収**:各成果物は PMO の機械検査(要件ID 網羅、規約検査、文書間突合)→ 内部レビュー → 発注者検収 の3段階。
2. **トレーサビリティ**:REQ/NFR → 要件定義書 → 基本・詳細設計書 → テスト仕様書 → IaC/コード のマトリクスを P1 で作成し全フェーズ更新。検収資料に毎回添付する。
3. **開発標準の機械的強制**(P2 で標準書化):linter、循環的複雑度10以内の CI ブロック、レイヤー依存方向検査(UI→アプリケーション→ドメイン→インフラの一方向)を CI に組み込み、違反ゼロを実装完了条件とする。
4. **IaC 品質**:steering/iac 規約に従い、静的検査(採用ツールに応じ cdk-nag / cfn-lint / tflint)エラー0、命名・必須タグ・セキュリティベースライン遵守、構成図⇔IaC 1:1 突合表を維持する。
5. **判断の文書化**:重要設計判断は ADR(02-deliverables/adr/、1判断1ファイル)なしに行わない。技術選定は P1 で3案以上の比較マトリクスによる技術選定理由書を作成する。

## 10. 成果物・文書管理

- 成果物一覧は RFP 第6章の表を正とし、P1 で文書管理計画書(記載粒度・様式・命名・改訂管理)を市と確定する。
