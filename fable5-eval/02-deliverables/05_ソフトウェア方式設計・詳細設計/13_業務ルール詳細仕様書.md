## 0. 本書の位置付け

1. 本書は、トレーサビリティマトリクス1.1版でP3割当としていた業務ルール詳細(REQ-009/010/015、および減免REQ-018・抽選REQ-008)の判定・計算仕様を定める。実装はすべて**ドメイン層**(`jp.lg.kasumidai.yoyaku.domain.*`)に配置し、単体テストで全ルールを検証する(KSM-TSP-001 §5.1)。
2. 数値の初期値(上限値・期限日数等)はすべて**マスタ設定値**であり、職員(所管課/管理者)が画面から施設別に変更できる。本書の初期値は現行運用(別紙1)からの当社案であり、**G3検収および業務部会での市確認をもって確定**する(確認一覧は§7)。

## 1. 予約上限ルール(REQ-009)

### 1.1 ルール構造

上限ルールは「施設×利用者区分」単位のマスタ(`reservation_limit_rules`)で定義し、次の4種を設定可能とする。

| ルール種別 | 内容 | 初期値(案) |
|---|---|---|
| L-1 月間コマ数上限 | 同一利用者が同一施設で同一利用月に保有できる確定・申込中コマ数の上限 | 体育施設:12コマ/月、公民館・文化系:8コマ/月 |
| L-2 同一日重複制限 | 同一利用者×同一日×同一施設の上限コマ数 | 連続3コマまで(同一日内) |
| L-3 同時保有件数 | 利用日が未到来の予約の同時保有上限(施設横断) | 30件 |
| L-4 予約受付開始日 | 利用者区分別の受付開始(市内:利用月の2か月前の1日 9:00/市外:1か月前の1日 9:00 等) | 区分マスタで定義(REQ-002) |

### 1.2 判定仕様

1. **カウント対象**:status ∈ {hold(仮押さえ), pending(確定待ち), confirmed(確定)} の予約明細。取消・期限切れは対象外。抽選申込(当落未確定)はカウントしないが、**当選確定時にL-1を再判定**し、超過分は当選対象から除外する(§5.4)。
2. **判定時点**:申込確定操作時(API)に、対象明細を含めた合算で判定。判定とINSERTは同一トランザクション(§2)。
3. **職員代行の特例**:窓口担当以上のロールは、上限超過時に警告表示の上で**上限超過登録(理由必須・操作ログ記録)**が可能(市主催事業・特例利用の運用に対応。REQ-021/022)。利用者本人のオンライン申込では特例不可。
4. 優先利用枠(REQ-022)内のコマは一般利用者の申込対象外(空き表示は「優先枠」表示)。

## 2. 一括予約の整合(REQ-010)

### 2.1 仕様

1. 一括予約(連続コマ・複数施設・定期利用)は**1予約(reservations)+複数明細(reservation_details)**として受け付け、**単一DBトランザクションで全件成立または全件不成立**とする(REQ-010の「全件成立/全件不成立」要件)。
2. 定期利用(毎週同一コマ等)は申込時に対象コマへ展開(展開上限:3か月先まで・最大26コマ/申込)し、一括予約と同一の処理とする。
3. 不成立時は、不成立コマと理由(予約済み/休館/優先枠/上限超過)の全件一覧を返却し(KSM-DDD-001 §4.4)、利用者は不成立コマを除外して再申込できる(部分自動確定は行わない:利用者の意図しない「歯抜け予約」と料金確定を防ぐ)。

### 2.2 同時実行制御(抽選初日・予約開始日のピークと整合)

1. 二重予約の最終防衛線は**部分一意インデックス**(`uq_active_slot`。KSM-DDD-001 §3.4):同時申込の競合は片方が一意制約違反でロールバックし、明確なエラー(409)を返す。
2. デッドロック防止:明細のINSERT順を(unit_id, use_date, slot_id)の昇順に正規化する。
3. 楽観方式を基本とし、行ロックの長時間保持を避ける(申込APIの処理は単一トランザクション・上限判定込みで100ms目標=NFR-B01余裕確保)。
4. 空き表示(60秒キャッシュ)と実在庫の差は申込時の再検証で吸収(KSM-ADR-009決定3)。

## 3. 料金算定(REQ-015)

### 3.1 算定式

```
明細料金 = 基本料金(施設×面・室×コマ×利用者区分) + Σ 付帯設備料金(設備×数量×コマ)
請求額   = Σ 明細料金 − 減免額(§4)
```

| 規則 | 仕様 |
|---|---|
| 料金解決 | fee_masterの「利用日時点で有効な版」(valid_from ≦ 利用日 < 次版valid_from)を適用。**申込時点ではなく利用日時点**の料金表を適用する(条例改定時の取扱いとして市確認事項=§7-3) |
| 市外加算 | 利用者区分(REQ-002)の料金区分で表現(例:市外=市内の1.5倍をマスタ値として保持。係数計算はせずマスタに実額を持つ:条例別表との突合を容易にするため) |
| 端数処理 | 料金マスタは円単位の実額を保持し、システムでの乗除算を行わない(端数規則の解釈差を排除)。減免による按分計算のみ円未満切捨て(§4.3) |
| 算定明細 | 内訳(適用料金表版・明細単価・設備・減免)をbillings.calculation_detail(JSONB)に保存し、帳票(RP-01/02)・監査で再現可能とする |
| キャンセル料 | 施設別の取消規則マスタ(取消期限・料率)による。初期値(案):利用日7日前まで無料/6日前〜前日50%/当日100%。還付額=収納済額−キャンセル料(§4.4・REQ-011/019) |

## 4. 減免(REQ-018)

### 4.1 減免区分(初期値案。条例・規則に基づき市確認=§7-4)

| 区分 | 内容 | 減免率 |
|---|---|---|
| 全額免除 | 市・教育委員会主催事業、市内学校の教育課程利用 等 | 100% |
| 半額減額 | 市内公益活動団体(登録団体)、障害者団体 等 | 50% |
| 個別決定 | 市長が特に認める場合 | 承認時に率を入力(0〜100%) |

### 4.2 ワークフロー

```
申請(利用者オンライン/窓口代行) → 受付(自動) → 審査(施設所管課) → 承認/否認 → 料金再計算 → 通知
```

1. 申請単位は予約(reservation)単位。申請時に証憑(登録団体証明等)の添付可(S3保存・暗号化)。
2. 承認権限:施設所管課ロール以上×担当施設のみ(REQ-023)。承認・否認とも理由を記録し操作ログへ(REQ-024)。
3. 申請中の支払期限は自動延伸(承認/否認確定+7日まで)。否認時は原額で請求継続。
4. 収納後の承認(遡及減免)は差額を還付対象(§4.4)として自動計上。

### 4.3 計算仕様

- 減免額 = 請求額(基本+設備)× 減免率、円未満切捨て。
- 複数減免区分の併用は不可。該当区分が複数ある場合は**利用者に最も有利な1区分**を適用(申請時に区分選択、審査時に変更可)。
- 減免はキャンセル料算定の基礎額にも適用(減免後額に対して料率適用)。

### 4.4 還付(REQ-019)

- 還付対象=取消時の収納済額−キャンセル料、および遡及減免の差額。還付方法(口座振込/窓口現金)と消込は市の出納ルールに従い、システムは対象一覧(RP-03)と状態管理(計上→支払依頼→完了)を提供する。オンライン決済の取消期間内はカード会社経由の取消(決済代行API)を優先する。

## 5. 抽選ロジック(REQ-008)

### 5.1 抽選サイクル(現行運用踏襲=別紙1)

| 項目 | 仕様(初期値=現行運用) |
|---|---|
| 対象 | 体育施設(体育館・テニス・グラウンド・野球場・武道館)。施設グループ単位に抽選期間マスタで設定可能(公民館等への将来拡大に対応) |
| 申込期間 | 毎月1日 9:00〜7日 23:59(翌々月分)※期間はマスタ設定 |
| 抽選実行 | 毎月8日 6:00(バッチJB-01)※利用の少ない早朝に実行し、9時の窓口開始までに結果確定 |
| 当選確定期限 | 当落通知から7日(支払期限=確定操作+施設別規則)。期限超過は自動取消(JB-03) |

### 5.2 申込仕様

1. 1申込=希望順位付きの希望コマ列(第1〜第3希望。各希望は連続コマ可)。
2. 同一利用者×同一抽選期間×同一施設グループの申込は**1件**(重複申込はDB一意制約で防止)。
3. 申込期間中は取下げ・希望変更可(締切後不可)。

### 5.3 抽選アルゴリズム(公平性担保)

```
入力: 抽選期間内の全申込(lottery_entries)
1. 各申込に暗号論的乱数(SecureRandom)による乱数キーを抽選実行時に付与
2. 乱数キー昇順に処理(申込時刻に依存しない=申込集中の不利を排除)
3. 各申込について希望順位順に空き判定:
   a. 希望コマ列の全コマが未割当 かつ 上限(§1)・休館・優先枠に抵触しない → 当選(該当希望で確定し以降の希望は評価しない)
   b. 全希望が不成立 → 落選(乱数キー順の落選順位を記録=繰上げ用)
4. 当選分を予約(pending・支払期限付き)として生成、当落通知をSQSへ投入
5. 実行サマリ(申込・当選・落選数、所要時間)を記録(JB-01)
```

- 乱数キー・処理順・判定結果を抽選実行ログとして保存し、**市が事後に抽選の公平性を検証可能**とする(議会・監査対応。REQ-025の趣旨)。
- グループ向け優遇・連続当選回避(過去当選者の優先度調整)は現行運用に存在しないため実装しない。将来要望時はマスタ追加で対応可能な構造とする(乱数キーへの重み付け拡張点をドメイン層に確保)。

### 5.4 当選時の上限再判定・繰上げ

1. 当選判定時に予約上限(§1)を再判定し、超過する希望は不成立として次希望を評価(複数施設グループの同時抽選でも上限整合を維持するため、抽選はグループ逐次実行)。
2. **繰上げ**:当選者の期限超過取消・自前取消が発生した場合、落選順位リスト(§5.3 3-b)に基づき**職員が画面(SC-S09)から繰上げを実行**する(自動繰上げは行わない。現行の窓口運用(別紙1「繰上げ運用」)との整合を優先し、誤繰上げ時の取消負担を避けるため。自動化はP4以降に市と運用確認の上、設定フラグで有効化可能な構造とする)。繰上げ時は対象者へ通知し、確定期限を再設定する。
