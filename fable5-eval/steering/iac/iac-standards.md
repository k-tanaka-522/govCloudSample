# IaC 規約(steering/iac)

本規約は本プロジェクトの IaC(Infrastructure as Code)成果物に適用する。
検査官はこの規約をチェックリストとして機械的に検査する。

## 1. ツールと構成
- IaC ツールは AWS CDK(TypeScript)または CloudFormation/Terraform とし、選定理由を ADR に記録すること。
- 環境差分(本番/検証/開発)はコード分岐ではなくパラメータ/コンテキストで管理すること。
- 手作業で構築する範囲がある場合、その範囲と理由を文書で宣言すること。

## 2. 命名・タグ
- リソース名は `{system}-{env}-{role}` 形式(例: `yoyaku-prod-app`)。env は `prod` / `stg` / `dev`。
- 全リソースに次のタグを必須とする:`Project`、`Env`、`ManagedBy`(IaCツール名)、`CostCenter`。

## 3. セキュリティベースライン
- S3 バケット:パブリックアクセスブロック有効、デフォルト暗号化(SSE-S3 以上)、バージョニングは用途に応じ判断し理由を記録。
- 保存データの暗号化:RDS・EBS・S3 すべて暗号化必須。鍵管理方式(AWSマネージド/CMK)は ADR で判断。
- 通信の暗号化:外部公開エンドポイントは TLS1.2 以上。HTTP は HTTPS へリダイレクト。
- セキュリティグループ:`0.0.0.0/0` のインバウンド許可は 443/80 のみ。SSH/RDP の全開放を禁止する。
- IAM:ワイルドカード(`Action: *` / `Resource: *`)の併用を禁止。最小権限とし、人とワークロードのロールを分離する。
- 監査ログ:CloudTrail 有効化を前提とし、アプリケーション・インフラのログ保管期間を設計書に明記する。

## 4. 品質ゲート
- 静的検査(cfn-lint / tflint / cdk-nag 等、採用ツールに応じたもの)をエラー0で通過すること。検査コマンドを README に記載する。
- 監視(アラーム・ダッシュボード)定義は IaC に含め、運用設計書の監視項目一覧と突合可能にすること。

## 5. 文書との突合
- 基本設計書の構成図に現れるリソースと IaC 実体は 1:1 に対応付け可能であること(対応表または命名規則による機械的突合)。
