# 設計書テンプレート規約：二軸構成の原則

## このファイルの位置づけ
`steering/design-templates/` は、受注エージェントが基本設計書・詳細設計書を
生成する際に従う「章立て」と「記述形式」を規定する共通コアである。
本ファイル（00）が原則、10が基本設計、20が詳細設計の対応表を定義する。
クライアント固有の差し替えは想定しない（行政受託の標準テンプレートとして共通コアに置く）。

## 中核原則：網羅性の軸と表現の軸を分離する

設計書の品質は、独立した二つの軸で決まる。この二つを混同しないこと。

- **網羅性の軸（章立て・トレーサビリティ・粒度）= IPA共通フレーム2013準拠を維持する。**
  どの章を立てるか、要件IDから設計・テストへどう追跡可能にするか、各章を
  どこまで書き込むかは、IPAの成果物体系を骨格とする。検収側（発注者・制度準拠検査官）も
  IPA基準で網羅性を見るため、ここは外さない。行政納品の前提条件である。

- **表現の軸（各章の記述形式・図法・記法）= 対象領域のデファクトを採用してよい。**
  UML一辺倒に固執しない。構成図・API仕様・データ構造・インフラ詳細・フロー表現は、
  下表（10・20）で指定する現代的記法に置き換える。これにより設計書とIaCコードが
  同じ語彙で書かれ、設計書⇔コードの突合検査が自然に成立する。

## 歯止め（モダン化を口実にした劣化の禁止）

1. **網羅性・トレーサビリティはIPAサンプルの粒度を下回らないこと。**
   記述形式を現代化しても、章の抜けや要件IDの追跡断絶があれば検収不可とする。
   「モダンに」を簡略化の言い訳にしない。

2. **章の器はIPA、章の中身の図法はモダン、という入れ子構造を守る。**
   章ごとIPA体系から差し替えることは禁止する。IPAの章立てを維持したまま、
   その章の内部の記述形式だけをデファクトに置き換える。これにより制度準拠検査官の
   IPA突合と、技術検査官の設計⇔コード突合の両方が通る。

3. **記述形式の選択には根拠を添える。**
   なぜその記法を採ったか（対象が何であるか）を各設計書の冒頭か該当章の注記に
   1行で示す。記法の選択自体が設計判断であり、必要に応じてADRを残す。

## 標準記述形式セット（行政受託・AWS/IaC前提）

本案件のデフォルトの表現軸は以下を標準とする（詳細は10・20の対応表で章ごとに割り当てる）。

| 用途 | 標準記法 | 補足 |
|---|---|---|
| システム全体・コンテナ・コンポーネント構成 | C4モデル（Context / Container / Component） | 構成図はC4の階層で。図はMermaid C4 または PlantUMLで版管理可能に |
| API仕様 | OpenAPI 3.x（YAML） | エンドポイント設計はOpenAPI記述を正本とし、設計書には要約表＋全文参照 |
| 非同期・イベント連携 | AsyncAPI | 採用する場合のみ |
| データ構造 | ER図 ＋ DDL / スキーマ定義（JSON Schema等） | ER図単独で終えず、物理スキーマと対にする |
| 処理フロー・状態遷移 | Mermaid（flowchart / sequenceDiagram / stateDiagram） | UMLアクティビティ図・シーケンス図の代替として可 |
| インフラ詳細設計 | IaCコード（CDK / Terraform）を設計の正とする | 構成図（C4 Container）＋IaC実体。設計書はIaCへの参照と意図の記述に徹する |
| 監視・運用 | SLI/SLO定義書式 ＋ IaC内のアラーム/ダッシュボード定義 | 運用設計書の監視項目とIaC実体が突合可能であること |

注：検収側がIPA基準で読むため、各設計書の冒頭に「本書はIPA共通フレーム2013の
成果物体系に準拠し、各章の記述形式のみ現代的記法を採用している」旨の凡例を置くこと。
