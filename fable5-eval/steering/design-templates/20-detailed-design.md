# 詳細設計書テンプレート：IPA章立て × モダン記述形式 対応表

本書は詳細設計フェーズの成果物構造を定義する。
左列はIPA共通フレーム2013のソフトウェア詳細設計に対応する章立てで固定。
右列は各章で採用する現代的記述形式。詳細設計は「コーディング可能な粒度」まで
落とすフェーズであり、記述形式はコードに最も近い記法を優先する。

凡例：記載粒度 ＝ 実装者がこの設計書だけで実装に着手できる下限。

| # | 章（網羅性の軸：IPA基準・固定） | 記述形式（表現の軸：採用してよいデファクト） | 記載粒度の目安（IPA下限） |
|---|---|---|---|
| 1 | はじめに・基本設計とのトレース | 基本設計の章ID対応表、参照ADR一覧 | 基本設計のどの章を詳細化したかを明示 |

## 詳細設計フェーズ固有の注意
- 3・4で状態の一貫性、9で例外網羅、8で冪等性——これらはESDR/ESPR系の
  設計観点（命題形式のレビュー項目）をチェックリスト化して観点列に流用すると、
  検査官の判定基準が機械化しやすい。
- 10・11・12をIaC実体と対にすることが、inspector-security の自己整合性検査
  （設計書の監視項目⇔IaCのアラーム定義の突合、ADR実在確認）の前提になる。
- 13・14のテスト観点→要件IDの紐付けが、inspector-consistency の逆順追跡の終端になる。

## 記法選択の早見（対象 → 記法）
- Web API中心 → OpenAPIを正本、処理はsequenceDiagram
- バッチ/ETL中心 → IPO＋flowchart、冪等性とリカバリを厚く
- 状態を持つ処理 → stateDiagramを必須化、一貫性保証を明記
- インフラ/IaC中心 → IaCコードを正、設計書は意図とC4 Componentに徹する

## モジュール粒度の原則（詳細設計・製造・単体テストの粒度統一）

### 原則：三工程はモジュール単位で粒度を揃える
詳細設計・製造（実装）・単体テストは、同一のモジュールIDを単位として粒度を揃える。
これにより、トレーサビリティの追跡単位がモジュールに固定され、検査官の逆順追跡
（単体テスト→詳細設計→基本設計→要件）がモジュールIDを背骨として機械的に成立する。

- 各モジュールに一意の **モジュールID（MOD-xxx）** を付番する。
- 1つのモジュールに対し、詳細設計・実装・単体テストが1対1対1で対応すること。
- モジュールIDは要件ID（REQ-xxx）と多対多で紐付き、後述のインデックス表で管理する。

### 判断事項：ファイル分割方針はADRに残す（決め打ちしない）
「1ファイル1モジュール」にするか「機能群単位で複数モジュールを1ファイルに束ねる」かは、
**規模に応じた判断**であり、テンプレートで一律に固定しない。受注者が次を根拠に決定し、
ADRに記録すること。

- モジュール数が小規模（目安20未満）：章で束ねて可読性を優先してよい。
- モジュール数が大規模（目安数十以上）：1ファイル1モジュールで管理性を優先する。
- いずれの場合も、分割方針と根拠（規模・チーム分業・レビュー単位）をADRに記す。
- 決定した分割方針は、製造・単体テストのファイル構成にも同じ単位で適用すること
  （詳細設計だけ細かく割り、実装は粗く束ねる、のような粒度のねじれを禁止する）。

### 必須成果物：モジュールインデックス兼トレーサビリティ表
ファイル数が増えるため、**モジュールID基点のインデックス表を正本として必須化する**。
この表は (a) ファイル管理の一覧、(b) 三工程のトレーサビリティマトリクス、
(c) 検査官の過不足照合の基準、の三役を一枚で兼ねる。
詳細設計テンプレ14章のトレーサビリティマトリクスは、この表に統合してよい。

配置：`02-deliverables/module-index.md`（または同等の機械可読な表）

| 列 | 内容 | 用途 |
|---|---|---|
| モジュールID | MOD-xxx（一意） | 三工程の追跡単位 |
| モジュール名 | 機能を表す名称 | 可読性 |
| 関連要件ID | REQ-xxx（複数可、カンマ区切り） | 要件への上方追跡 |
| 詳細設計ファイル | パス（章束ねの場合は章アンカーまで） | 設計実体への参照 |
| 製造ファイル | 実装ファイルのパス | コード実体への参照 |
| 単体テストファイル | テストファイルのパス | テスト実体への参照 |
| 状態 | 設計済 / 実装済 / テスト済 / 検収済 | 進捗と完成度 |
| 備考 | 分割方針の例外・依存関係等 | 判断の痕跡 |

注意：このインデックスを正本にすると「インデックスを更新し忘れ、実体とズレる」事故が
起きる（現実の開発でも頻発する）。**ズレの検出自体が検査官の見せ場**であり、欠陥ではなく
評価ポイントとして扱う。受注者はインデックスと実体の同期を実装完了の条件とすること。

### 検査官との連動（双方向突合）
inspector-consistency の検査手順に、本インデックスとの双方向突合を加える：
- **順方向**：インデックスに記載された全ファイル（設計・実装・テスト）が実在するか。
- **逆方向**：02-deliverables/ 配下の実ファイルが、すべてインデックスに登録されているか。
- **粒度整合**：同一モジュールIDに対し、詳細設計・製造・単体テストが揃っているか
  （いずれかの工程が欠けているモジュールを検出する）。
片方向のみ（インデックス→実体）だと、登録漏れのファイルが検出できないため、
必ず両方向で照合すること。
