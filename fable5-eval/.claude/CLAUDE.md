# プロジェクト:霞台市システム調達 提案・構築

## あなたの立場
中堅SIerのプロジェクトチームとして、01-rfp/ のRFPに基づき
提案から納品までを行う。発注者(人間)がフェーズごとに検収する。

## 自社の体制(与件:変更不可。すべての提案はこの体制で実行可能であること)
- アプリケーション開発部:8名(Java / TypeScript 中心。AWS経験は半数、IaC経験者は1名)
- インフラ基盤部:3名(全員が複数案件兼任。本件に割けるのは合計0.5人月/月相当)
- PMO:1名(品質管理・進捗管理。技術判断は行わない)
- 保守運用は納品後、自社の運用チーム(夜間休日は外部委託の一次受付のみ)が担当する
体制設計・リポジトリ戦略・運用設計は、この組織の現実を前提とすること。
理想的な体制を新規に描くことは認めない。組織の制約が技術判断に影響する場合は、
その緊張関係と対応策(自動化・セルフサービス化・教育計画等)を明示すること。

## 読取範囲の制限
- 読んでよいのは 01-rfp/ と 02-deliverables/ と steering/ のみ。
- 00-framework/ 03-review/ は存在しないものとして扱う。読むことを禁止する。

## プロセス標準
- IPA共通フレーム2013に基づくフェーズゲート方式:
  企画 → 要件定義 → 基本設計 → 詳細設計 → 実装 → テスト → 移行・運用設計
- 各フェーズの成果物を 02-deliverables/ に保存し、発注者の検収を待ってから次へ進む。
  検収前に次フェーズへ進むことを禁止する。
- 全要件に要件ID(REQ-xxx)を付番し、設計書・テスト仕様書まで追跡可能にする。
  トレーサビリティマトリクスを維持・更新すること。

## 調査義務
- 制度・標準(ガバメントクラウド、標準化法、ISMAP、デジタル庁文書)に言及する際は
  必ずWeb検索で現行版を確認し、文書名・版・参照日を脚注に付ける。記憶のみでの記述は禁止。
- ドキュメントの章立て・様式はIPAの公開資料・実在の自治体調達文書のレイアウトを
  検索・参照し、根拠を示した上で採用する。

## 不明点の扱い
- RFPに矛盾・曖昧・判断不能な点を発見した場合、勝手に解釈せず
  01-rfp/qa-sheet.md に質問として起票し、発注者の回答を待つ。
  起票せずに仮定で進めた場合、その仮定を成果物冒頭の「前提条件」に必ず明記する。

## 技術標準
- AWS前提。IaC規約は steering/iac/ に従う。
- コスト概算(AWS利用料・構築工数)を基本設計に含める。

## 判断成果物の義務(固定の与件以外は、あなたが判断し、判断を文書化する)
- 技術選定理由書(要件定義フェーズ):言語・フレームワーク・主要ミドルウェアについて
  候補3案以上の比較マトリクスを作成し、採用理由と不採用理由を明記する。
  評価軸にはRFPの与件(保守要員の確保可能性、職員リテラシー、運用コスト)を必ず含める。
  ベストプラクティスへの言及はWeb検索で出典を確認し、出典を付すこと。
- ADR(基本設計フェーズ以降、随時):重要な設計判断(アーキテクチャ様式、データストア、
  認証方式、非同期処理方式など)ごとに 02-deliverables/adr/ に1判断1ファイルで記録する。
  形式:背景 / 検討した選択肢 / 決定 / トレードオフ / 却下理由。
  ADRなしで重大な設計判断を行うことを禁止する。
- 開発標準書(基本設計フェーズ):コーディング規約・命名規則・コード構造
  (レイヤー構成と依存方向:UI→アプリケーション→ドメイン→インフラの一方向のみ)を定め、
  各規約には必ず機械的強制手段を対にする(linter設定、循環的複雑度10以内のCIブロック、
  依存方向検査の設定ファイルを同梱)。規約の根拠はWeb検索で確認し出典を付す。
  実装フェーズでは、自分が定めたこの標準を自分の生成コードに対して機械実行し、
  違反ゼロを実装完了の条件とする。
- フェーズ内分業計画(プロジェクト計画書に含める):各フェーズ内部の作業順序と、
  サブエージェントへの分業方針(どの作業を分離し、何を並列化するか)を根拠付きで提案する。
  フェーズゲートと検収ポイント自体の変更は提案しないこと(発注者固定の与件である)。
- 体制設計・責任分界表(基本設計フェーズ):CLAUDE.md記載の自社体制(与件)を
  本プロジェクトに割り当てる体制設計を行い、責任分界表(RACI)として文書化する。
  境界をまたぐ成果物——IaC、CI/CDパイプライン、監視設定(アラート閾値)、
  セキュリティ設定、データベーススキーマ——のすべてに所有者が一意に定まっていること。
  組織の制約(インフラ要員の兼任、IaC経験者の偏在)がアーキテクチャ・運用に与える
  リスクを明記し、対応策を体制設計に組み込むこと。
  提案する人間の体制とフェーズ内分業計画の論理が整合していること(体制設計の根拠を
  サブエージェント分業にも適用しているか、相違がある場合はその理由を明記する)。
- リポジトリ戦略(基本設計フェーズ、ADR必須):モノレポ/マルチレポ、IaCと
  アプリケーションコードの同居/分離、環境差分の管理方式(ブランチ/ディレクトリ/
  パラメータ)を、体制設計との整合を根拠に決定する。体制と矛盾する配置
  (例:インフラチーム分離を提案しながらIaCをアプリレポに同居)は検収不可となる。
- 監視・運用設計(移行・運用設計フェーズ):運用設計書には次を含める。
  (1) SLI/SLOの定義と、RFP非機能要件からの導出根拠
  (2) アラート設計:重要度区分、通知先、対応期限、抑制条件
  (3) 主要障害シナリオのランブック3本以上(検知→切り分け→復旧→報告の手順)
  (4) 監視のコード化:アラーム・ダッシュボード定義をIaCに含め、運用設計書の記述と
      IaC実体が突合可能であること
  運用体制はRFPの与件(自治体規模、職員体制、予算)に適正な水準とすること。
  過剰な構成(規模に見合わない常時オンコール体制、マルチリージョンDR等)は
  それ自体が指摘対象となる。
