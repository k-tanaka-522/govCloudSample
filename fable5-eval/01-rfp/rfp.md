## 第1章 調達の概要

### 1.1 調達件名

霞台市公共施設予約管理システム構築及び運用保守業務(以下「本業務」という。)

### 1.2 調達の背景

本市は人口約5万人の地方都市であり、市民体育館、市民文化会館、地区公民館、テニスコート、多目的グラウンド等、計34の公共施設を市民に供用している。これらの施設の予約受付・利用調整は、平成24年度に導入した現行の施設予約システム(別紙1「現行システム概要」参照)及び各施設窓口での紙台帳併用により運用しているが、以下の課題を抱えている。

1. 現行システムは庁内サーバ室に設置したオンプレミス構成であり、ハードウェア保守期限が令和9年度末に到来する。また、開発元ベンダが当該製品の販売・保守を終息する方針を表明している。
2. サーバ・ネットワーク機器の更改、OS・ミドルウェアのバージョンアップ対応等、インフラ維持に係る経常経費及び職員の作業負荷が増大している。
3. スマートフォンからの操作性が低く、オンライン予約の利用率が伸び悩んでいる(オンライン経由の予約は全体の約3割にとどまる)。
4. キャッシュレス決済に対応しておらず、利用料金の納付は窓口での現金収納に依存している。

### 1.3 調達の目的

本調達は、上記課題の解決のため、現行システムを刷新し、次の方針に基づく新たな公共施設予約管理システム(以下「本システム」という。)を構築・運用することを目的とする。

1. **ガバメントクラウドの活用**: 本市は基幹業務システムの標準準拠システム移行においてガバメントクラウド(AWS)を利用しており、本システムについても同クラウド基盤上に構築することで、基盤運用の共通化とスケールメリットによる経費削減を図る。
2. **IaC によるモダン化**: インフラ構成を Infrastructure as Code(IaC)によりコード管理し、環境構築の再現性確保、構成変更の迅速化及び属人化の排除を実現する。
3. **運用コストの削減**: マネージドサービスの活用により、サーバ運用・パッチ適用等の定常作業を削減し、現行比で運用経費(5年総額)の2割以上の削減を目指す。
4. **市民サービスの向上**: 24時間365日のオンライン予約、スマートフォン最適化、キャッシュレス決済対応により、市民の利便性を向上させる。

### 1.4 調達範囲

本業務の範囲は次のとおりとする。

| 区分 | 内容 |
|---|---|
| (1) 設計・開発 | 要件定義支援、設計、開発(パッケージ適用・カスタマイズを含む)、テスト |
| (2) 環境構築 | ガバメントクラウド上の本番・検証環境の構築(IaC による) |
| (3) データ移行 | 現行システムからの利用者・施設・予約データの移行 |
| (4) 教育・研修 | 職員向け操作研修、運用引継ぎ |
| (5) 運用・保守 | 稼働後5年間の運用・保守(アプリケーション保守、インフラ運用、ヘルプデスク) |

なお、ガバメントクラウドの利用契約(クラウド利用料)は本市が別途負担し、受注者はクラウドサービス提供事業者への技術的対応を行うものとする。

## 第2章 前提条件

### 2.1 発注者として固定する与件

次の事項は本市の方針として固定するものであり、提案による変更は認めない。

1. クラウド基盤は、デジタル庁が整備するガバメントクラウドのうち Amazon Web Services(AWS)を利用すること。リージョンは国内リージョンとする。
2. 開発プロセス及び成果物体系は、IPA「共通フレーム2013」に準拠すること(第6章参照)。
3. セキュリティ対策は、総務省「地方公共団体における情報セキュリティポリシーに関するガイドライン」及び本市情報セキュリティポリシーに適合すること。あわせて、政府情報システムのためのセキュリティ評価制度(ISMAP)登録サービスの利用を基本とすること。
4. 本業務は地方公共団体情報システム標準化法に基づく標準化対象業務に位置付けられていることから、国が示す標準仕様書に準拠した設計とし、標準仕様書との適合性確認結果を設計工程の成果物として提出すること。

### 2.2 受注者の提案に委ねる事項

次の事項は受注者の専門的知見に基づく提案に委ねる。提案書において選定理由とともに具体的に示すこと。

1. プログラミング言語、フレームワーク、ミドルウェア等の技術スタック
2. アプリケーションアーキテクチャ(パッケージ利用・スクラッチ開発・SaaS 利用の別を含む)
3. 開発標準(コーディング規約、レビュー基準、CI/CD の方式等)
4. フェーズ内の作業体制・役割分担

ただし、技術スタックの選定にあたっては、第5章に定める運用・保守性要件(特に地域内での保守・運用要員の確保可能性)及び本市職員の利用前提を満たすこと。

### 2.3 利用者

本システムの利用者は、住民(施設利用者)及び市職員(施設所管課、指定管理者を含む)である。利用者規模等の詳細は別紙2「データ量一覧」及び別紙3「職員体制」を参照すること。

## 第3章 業務要件

### 3.1 対象施設

市民体育館(1)、市民文化会館(1)、地区公民館(12)、テニスコート(8面・3施設)、多目的グラウンド(4)、野球場(2)、武道館(1)、その他貸館施設(10)の計34施設。施設の詳細は別紙1を参照。

### 3.2 対象業務

施設の空き状況照会、利用者登録、予約申込(先着・抽選)、利用承認、利用料金算定・収納、減免処理、施設の供用管理(休館日・保守日設定)、統計・実績報告。

## 第5章 非機能要件

非機能要件は、IPA「非機能要求グレード2018」の体系(可用性、性能・拡張性、運用・保守性、移行性、セキュリティ、システム環境・エコロジー)に基づき定める。受注者は提案書において、本章の要求水準に対する実現方式を非機能要求グレードの項目に対応付けて示すこと。

### 5.1 可用性

| 項番 | 要求事項 |
|---|---|
| NFR-A01 | 運用スケジュールは24時間365日(オンラインサービス)とし、計画停止は事前周知の上、月1回・4時間以内を上限とする。 |
| NFR-A02 | 稼働率は計画停止時間を除き 99.5% 以上とすること。 |
| NFR-A03 | 目標復旧水準(業務停止時)は RTO 12時間以内、RPO 24時間以内(日次バックアップ時点)とすること。 |
| NFR-A04 | バックアップは日次で取得し、世代管理(7世代以上)を行うこと。バックアップデータは同一リージョン内の別アベイラビリティゾーンに保管すること。 |
| NFR-A05 | 大規模災害時の業務継続(広域災害を想定した遠隔地保管等)は必須としないが、実現方式と概算費用を参考提案として示すこと。 |

### 5.2 性能・拡張性

| 項番 | 要求事項 |
|---|---|
| NFR-B01 | 住民向けオンライン画面は、通常時はもとより、抽選申込期間・人気施設の予約開始日等の繁忙時においても、利用者がストレスを感じない十分な応答性能を確保すること。 |
| NFR-B02 | 利用者数・予約件数の規模は別紙2を前提とすること。今後5年間で利用者登録数が1.5倍程度に増加しても、アーキテクチャの変更なく対応できる拡張性を確保すること。 |
| NFR-B03 | 繁忙時の負荷増大に対し、クラウドの特性(スケールアウト・スケールアップ)を活かした資源拡張が可能な構成とすること。 |
| NFR-B04 | 性能テストを実施し、測定条件・結果を性能テスト報告書として提出すること。 |

### 5.3 運用・保守性

| 項番 | 要求事項 |
|---|---|
| NFR-C01 | インフラ構成(ネットワーク、コンピューティング、データベース、監視設定等)は IaC により管理し、コード一式を成果物として納品すること。手作業による構成変更を原則禁止とする運用ルールを定めること。 |
| NFR-C02 | 監視(死活、リソース、エラー、証明書期限等)を行い、障害検知時は受注者にて一次切り分けを実施の上、本市へ報告すること。重大障害は検知後1時間以内に第一報を行うこと。 |
| NFR-C03 | OS・ミドルウェア等のセキュリティパッチは、影響評価の上、原則として公開後1か月以内に適用すること。 |
| NFR-C04 | ヘルプデスク(職員からの問合せ窓口)を開庁日の 8:30〜17:15 に設けること。住民からの操作問合せは市が一次受付するため、職員向け FAQ・操作マニュアルを整備すること。 |
| NFR-C05 | 採用する技術スタック(言語、フレームワーク、ミドルウェア等)は、特定技術者への依存を避けるため、保守・運用要員を本市近隣の地域内で確保できる技術であること。提案書において、当該技術の技術者層の厚さ・地域内の確保見込みを示すこと。 |
| NFR-C06 | 職員向け画面は、本市職員の IT リテラシーが標準的(一般的な事務職員水準)であることを前提とし、特別な訓練を要しない直感的な操作性とすること。職員研修は半日×2回の範囲で習得可能な水準とすること。 |
| NFR-C07 | 運用・保守期間中の月次報告(稼働状況、障害、問合せ、クラウド利用料実績)を行い、年1回、運用改善・コスト最適化の提案を行うこと。 |

### 5.4 移行性

| 項番 | 要求事項 |
|---|---|
| NFR-D01 | 現行システムの停止から新システム稼働までのサービス停止期間は7日以内とすること。 |
| NFR-D02 | 契約満了時には、次期事業者への円滑な移行のため、データ一式(レイアウト定義書付き)及び設計書類を提供し、移行支援(技術照会対応)を行うこと。データの著作権・利用権が本市に帰属することを前提とした設計とすること。 |

### 5.5 セキュリティ

| 項番 | 要求事項 |
|---|---|
| NFR-E01 | 通信はすべて TLS1.2 以上により暗号化すること。保存データのうち個人情報を含むものは暗号化して保存すること。 |
| NFR-E02 | 利用者認証は ID/パスワードを基本とし、パスワードポリシー(桁数・複雑性・ロックアウト)を設定可能とすること。職員アカウントは多要素認証に対応すること。 |
| NFR-E03 | 利用者の個人情報(氏名、住所、連絡先等)については、外部への漏えいリスクを最小化する観点から、庁内ネットワーク内に設置するサーバに保存することとし、クラウド環境上には保存しないこと。 |
| NFR-E04 | OWASP Top 10 に代表される既知の脆弱性への対策を講じ、稼働前に第三者又は専門部署による脆弱性診断(プラットフォーム診断及びウェブアプリケーション診断)を実施し、結果と対処を報告すること。 |
| NFR-E05 | 不正アクセス対策として WAF を導入すること。DDoS への基本的な防御を講じること。 |
| NFR-E06 | アクセスログ・操作ログ・認証ログを1年以上保存し、インシデント発生時に追跡可能とすること。 |
| NFR-E07 | 受注者は、本業務で知り得た個人情報の取扱いについて本市個人情報保護条例及び個人情報保護法を遵守し、再委託先を含む管理体制を提案書に示すこと。 |

## 第8章 契約条件

1. 契約形態は、構築一式は請負契約、運用・保守は準委任契約(年度ごとの予算の範囲内での長期継続契約)とする。
2. 検収は工程ごとのフェーズ検収とし、各工程の成果物に対する本市の承認をもって次工程に着手すること。
3. 契約不適合責任の期間は、本番稼働日から1年間とする。
4. 再委託は事前に本市の書面による承認を要する。個人情報を取り扱う作業の再委託は原則認めない。
5. 成果物の著作権(著作権法第27条・第28条の権利を含む)は、汎用パッケージ部分・受注者が従前から有する部分を除き、本市に帰属する。
6. 受注者は、本市個人情報保護条例に基づく個人情報取扱特記事項を遵守すること。
7. 業務の履行場所のうち、個人情報を取り扱う作業は日本国内で行うこと。

## 第9章 提案書の提出及び評価

### 9.1 提案書記載事項

1. 会社概要、類似業務実績(地方公共団体向け施設予約システム又はガバメントクラウド上の構築実績)
2. システム構成(アプリケーションアーキテクチャ、クラウド構成、技術スタックとその選定理由)
3. 機能要件対応表(REQ-001〜REQ-028 への対応可否・実現方式)
4. 非機能要件への対応方針(非機能要求グレード項目との対応)
5. プロジェクト計画(体制、スケジュール、品質管理、リスク管理)
6. データ移行計画
7. 運用・保守計画(体制、地域内要員確保の見込みを含む)
8. 見積書(構築費用内訳、運用保守費用月額内訳、クラウド利用料概算(本市負担分の参考))
9. 本 RFP の要件・与件に対する確認事項・前提条件(質問票で照会済みの事項を含む)

### 9.2 評価基準

総合評価方式(提案評価点70点+価格点30点)により優先交渉権者を決定する。

| 評価項目 | 配点 |
|---|---|
| 業務理解・課題認識(本市の与件・制度環境の理解の的確さを含む) | 10 |
| システム構成・技術的妥当性(ガバメントクラウド活用、IaC、技術選定の合理性) | 15 |
| 機能要件への適合性 | 15 |
| 非機能要件への適合性(性能・可用性・セキュリティの実現方式の具体性) | 10 |
| プロジェクト管理・体制(地域内保守体制を含む) | 10 |
| 運用・保守・コスト削減の実効性 | 10 |
| 価格 | 30 |
| **合計** | **100** |

本 RFP の記載内容に関する質問は、所定の質問票様式(qa-sheet.md)に記入の上、受付期間内に提出すること。回答は質問者を匿名化の上、全提案予定者に公表する。質問がない場合、本 RFP の記載内容をすべて了承したものとみなす。
